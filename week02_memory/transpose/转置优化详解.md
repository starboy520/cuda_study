## 一、统一约定

代码位置：`transpose.cu` 第 15–25 行

```cuda
// ---------------------------------------------------------------------------
// 统一约定（与 mat_mul / vec_add 一致）
//
//   threadIdx.x / blockIdx.x  →  列 col（横向）
//   threadIdx.y / blockIdx.y  →  行 row（纵向）
//
//   行主序索引：matrix[row * width + col]
//
//   转置：out[col][row] = in[row][col]
//         out[row_out * height + col_out] = in[row_in * width + col_in]
// ---------------------------------------------------------------------------

constexpr int TILE = 32;
```

### 逐条解释

| 符号 | 含义 |
|------|------|
| `width` | 输入矩阵 `in` 的**列数** |
| `height` | 输入矩阵 `in` 的**行数** |
| `in[row * width + col]` | 行主序：第 `row` 行、第 `col` 列的元素 |
| 转置后 | `in` 是 `height×width`，`out` 是 `width×height`（行列互换） |
| `TILE = 32` | 每个 block 处理 32×32 的小块；也是 block 的 x、y 维度 |

### 矩阵尺寸关系

```text
输入 in：  height 行 × width 列
输出 out： width 行 × height 列

例：in 是 1024×768 → out 是 768×1024
```

### 转置下标（核心公式，后面会反复用到）

```text
in[row_in][col_in]  转置后变成  out[col_in][row_in]

用行主序线性地址写：
  读：in[row_in * width + col_in]
  写：out[row_out * height + col_out]
      其中 row_out = col_in，col_out = row_in
```

## 二、Host 启动配置

代码位置：`transpose.cu` 第 119–120 行（在 `run_kernel` 里）

```cuda
dim3 block(TILE, TILE);   // block 是 32×32 = 1024 个线程
dim3 grid((width + block.x - 1) / block.x,
          (height + block.y - 1) / block.y);
kernel<<<grid, block>>>(d_in, d_out, width, height);
```

### 解释

```text
block(TILE, TILE)  →  blockDim.x=32, blockDim.y=32
                      每个 block 里 1024 个线程，排成 32×32 方阵

grid.x = ceil(width / 32)   →  横向盖多少块（按 in 的列切）
grid.y = ceil(height / 32)  →  纵向盖多少块（按 in 的行切）

blockIdx.x 走 in 的「列方向」
blockIdx.y 走 in 的「行方向」
```

### 一个线程的全局坐标（读 in 时的「标准公式」）

```cuda
col = blockIdx.x * blockDim.x + threadIdx.x;   // x → 列
row = blockIdx.y * blockDim.y + threadIdx.y;   // y → 行
```

### Warp 是什么（理解「合并读写」的关键）

```text
一个 warp = 32 个线程，硬件绑在一起执行

block(32,32) 里，通常 threadIdx.y 相同、threadIdx.x = 0..31 的 32 个线程
组成一个 warp（同一「行」上的 32 个线程）

合并访问：这 32 个线程若访问 global memory 里连续的 32 个 float
         → GPU 合并成少量内存事务 → 快
不合并：  32 个线程访问地址间隔很大 → 32 次独立访问 → 慢
```

## 三、版本 1：`transpose_naive`

代码位置：`transpose.cu` 第 29–40 行

```cuda
// 版本 1：只用 global memory（读合并、写不合并，作对比基线）
__global__ void transpose_naive(const float* in, float* out, int width,
                                int height) {
  const int col = blockIdx.x * blockDim.x + threadIdx.x;  // x → 列
  const int row = blockIdx.y * blockDim.y + threadIdx.y;  // y → 行

  if (row < height && col < width) {
    const int in_idx = row * width + col;
    const int out_idx = col * height + row;  // out 宽=height
    out[out_idx] = in[in_idx];
  }
}
```

### 逐行解释

**第 32–33 行：算这个线程负责 in 的哪个元素**

```text
col = blockIdx.x * 32 + threadIdx.x
row = blockIdx.y * 32 + threadIdx.y

每个线程固定处理 in[row][col] 这一个数
```

**第 35 行：边界检查**

```text
矩阵不一定是 32 的整数倍，最后一个 block 会多出「无效线程」
if (row < height && col < width) 防止越界
```

**第 36–37 行：线性地址**

```text
in_idx  = row * width + col     ← in 里读哪里
out_idx = col * height + row    ← 转置后写哪里
         = row_out * height + col_out
         其中 row_out=col, col_out=row
```

**第 38 行：一行完成转置**

```text
out[col][row] = in[row][col]
```

### 性能问题

```text
【读 in】同一 warp：row 相同，col = 0,1,2,...,31 连续
        → in 地址连续 → 合并读 ✓

【写 out】同一 warp：col 连续，但 out_idx = col * height + row
        → 相邻线程地址差 height 个 float → 不合并 ✗
```

这是后面引入 shared memory 的原因。

## 四、版本 2a：`transpose_shared_simple`

代码位置：`transpose.cu` 第 42–60 行

```cuda
// 版本 2a：shared memory，同一线程读/写（最易理解，正确性优先）
__global__ void transpose_shared_simple(const float* in, float* out, int width,
                                        int height) {
  __shared__ float tile[TILE][TILE + 1];

  const int col = blockIdx.x * blockDim.x + threadIdx.x;
  const int row = blockIdx.y * blockDim.y + threadIdx.y;

  // 阶段 1：读 in[row][col] → shared
  if (row < height && col < width) {
    tile[threadIdx.y][threadIdx.x] = in[row * width + col];
  }
  __syncthreads();

  // 阶段 2：同一 (row,col) 写到 out[col][row]
  if (row < height && col < width) {
    out[col * height + row] = tile[threadIdx.y][threadIdx.x];
  }
}
```

### 第 45 行：`__shared__ float tile[TILE][TILE + 1]`

```text
tile 是 block 内所有线程共享的「小仓库」，存在 SM 的 shared memory 里
大小 32×33（+1 是 padding，版本 2b 写阶段会用到，这里先声明上）

每个 block 有自己独立的一份 tile，block 之间不共享
```

### 阶段 1（第 50–53 行）：global → shared

```cuda
tile[threadIdx.y][threadIdx.x] = in[row * width + col];
```

```text
threadIdx.x → tile 的列下标
threadIdx.y → tile 的行下标

和读 in 一样：warp 内 x 连续 → 读 in 合并 ✓
```

**为什么要 `__syncthreads()`（第 54 行）？**

```text
必须等 block 内 1024 个线程都把数据搬进 tile，才能开始写 out
否则有的线程还在读，有的已经开始写 → 数据错乱
```

### 阶段 2（第 56–59 行）：shared → global

```cuda
out[col * height + row] = tile[threadIdx.y][threadIdx.x];
```

```text
同一个线程：读 in[row][col]，写 out[col][row]
shared 只是中转，逻辑和 naive 完全一样 → 一定正确

但写 out 的地址公式没变 → 写仍然不合并 ✗
```

### 和 naive 比多了什么、解决了什么

| | naive | shared_simple |
|---|-------|---------------|
| 读 in | 合并 ✓ | 合并 ✓ |
| 写 out | 不合并 ✗ | 不合并 ✗ |
| 正确性 | ✓ | ✓ |
| 理解难度 | 低 | 低（多了 shared 中转） |

shared_simple 的价值：**先理解 shared memory 用法**，不追求写合并。

## 五、版本 2b：`transpose_shared`（重点）

代码位置：`transpose.cu` 第 62–82 行

```cuda
// 版本 2b：shared memory + block 映射交换（写回也合并，Week 2 优化版）
__global__ void transpose_shared(const float* in, float* out, int width,
                                 int height) {
  __shared__ float tile[TILE][TILE + 1];

  const int col_in = blockIdx.x * blockDim.x + threadIdx.x;
  const int row_in = blockIdx.y * blockDim.y + threadIdx.y;

  if (row_in < height && col_in < width) {
    tile[threadIdx.y][threadIdx.x] = in[row_in * width + col_in];
  }
  __syncthreads();

  // 写回时交换 block 内 x/y 映射，使 global write 合并
  const int col_out = blockIdx.y * blockDim.y + threadIdx.x;
  const int row_out = blockIdx.x * blockDim.x + threadIdx.y;

  if (row_out < width && col_out < height) {
    out[row_out * height + col_out] = tile[threadIdx.x][threadIdx.y];
  }
}
```

下面分**读阶段**和**写阶段**详细讲。

---

### 5.1 读阶段（第 67–73 行）——和 shared_simple 完全相同

```cuda
const int col_in = blockIdx.x * blockDim.x + threadIdx.x;
const int row_in = blockIdx.y * blockDim.y + threadIdx.y;

if (row_in < height && col_in < width) {
  tile[threadIdx.y][threadIdx.x] = in[row_in * width + col_in];
}
__syncthreads();
```

```text
变量改名 col_in / row_in，强调这是「输入矩阵」的坐标
逻辑与 shared_simple 阶段 1 一模一样 → 合并读 ✓
```

---

### 5.2 写阶段（第 75–81 行）——和 shared_simple 不同，重点！

```cuda
const int col_out = blockIdx.y * blockDim.y + threadIdx.x;
const int row_out = blockIdx.x * blockDim.x + threadIdx.y;

if (row_out < width && col_out < height) {
  out[row_out * height + col_out] = tile[threadIdx.x][threadIdx.y];
}
```

#### 对比：shared_simple vs shared 的写回

| | shared_simple | shared（优化） |
|---|---------------|----------------|
| 输出 row | `col = bx*32+tx` | `row_out = bx*32+ty` |
| 输出 col | `row = by*32+ty` | `col_out = by*32+tx` |
| tile 取值 | `tile[ty][tx]` | `tile[tx][ty]` |
| 写 out 合并 | ✗ | ✓ |

**注意：写 out 时 threadIdx.x / y 的「角色」和读 in 时对调了。**

---

### 5.3 为什么要 `col_out = blockIdx.y * 32 + threadIdx.x`？

目标：**一个 warp 里 32 个线程写 out 的同一行、相邻 32 列**。

```text
out 行主序地址 = row_out * height + col_out

要连续，需要：
  同一 warp 内 row_out 相同
  同一 warp 内 col_out = 0,1,2,...,31 连续
```

一个 warp 里 `threadIdx.y` 相同、`threadIdx.x = 0..31`：

```cuda
row_out = blockIdx.x * 32 + threadIdx.y;   // y 相同 → row_out 相同 ✓
col_out = blockIdx.y * 32 + threadIdx.x;   // x 连续 → col_out 连续 ✓
```

所以 **`threadIdx.x` 在这里不是「表示列」的语义，而是「让 warp 写连续地址」**。

---

### 5.4 为什么要 `tile[threadIdx.x][threadIdx.y]`（下标对调）？

写回时分工换了，如果还取 `tile[ty][tx]` 会写错数。

**4×4 例子，block(0,0) 左上角 2×2：**

读完后 shared 里：

```text
tile[0][0]=a  tile[0][1]=b
tile[1][0]=e  tile[1][1]=f
```

转置后 out 里这块应是：

```text
  a  e
  b  f
```

写 out 第 0 行（a, e 挨着）——线程 (tx=0,ty=0) 和 (tx=1,ty=0)：

```text
row_out = 0+0 = 0,  col_out = 0+0=0  →  写 out[0]=?  需要 a  →  tile[0][0] = tile[tx][ty] ✓
row_out = 0+0 = 0,  col_out = 0+1=1  →  写 out[1]=?  需要 e  →  tile[1][0] = tile[tx][ty] ✓
```

读用 `tile[ty][tx]`，写用 `tile[tx][ty]` → 在 shared 里相当于**转 90°**。

---

### 5.5 手工验证（32×32 block 内，bx=0, by=0，线程 tx=5, ty=3）

**读阶段：**

```text
col_in = 0*32+5 = 5
row_in = 0*32+3 = 3
tile[3][5] = in[3*width+5] = in[3][5]
```

**写阶段：**

```text
row_out = 0*32+3 = 3
col_out = 0*32+5 = 5
out[3*height+5] = tile[5][3]
```

`tile[5][3]` 是读阶段线程 (tx=3, ty=5) 写入的：

```text
tile[5][3] = in[5][3]
```

转置：in[5][3] → out[3][5] ✓

---

### 5.6 blockIdx 在读/写中的角色

```text
读 in：block (blockIdx.x, blockIdx.y) 负责 in 里的一块
       blockIdx.x → 列方向，blockIdx.y → 行方向

写 out：转置后这块要放到「行列对调」的位置
       blockIdx.x 原来管 in 的列 → 现在参与 row_out
       blockIdx.y 原来管 in 的行 → 现在参与 col_out
       再配合 threadIdx.x/y 对调
```

可以先忽略 blockIdx，只记 4×4 例子里「warp 合写一行 + tile 下标对调」。

---

### 5.7 `tile[TILE + 1]` 的 +1（bank conflict）

```text
写阶段：32 个线程读 tile[threadIdx.x][threadIdx.y]，threadIdx.x 各不相同

若第二维 stride = 32：
  tile[0][y], tile[1][y], ... tile[31][y] 可能落在同一 shared memory bank
  → 32 路 bank conflict → 慢

第二维改成 33（+1 padding）：
  每行错开 1 个 float → bank 错开 → 无冲突
```

---

### 5.8 一张图串起来（一个 warp）

```text
【读 in】threadIdx.x 往右扫 → 读 in 的一行
        a ── b ── c ── d ...
        ↓ tile[ty][tx]

【写 out】同一 warp 横向排队 → 写 out 的一行
        a ── e ── i ── m ...
        ↑  ↑
     tx=0 tx=1   从 tile[tx][ty] 取
```

## 六、4×4 手工推演

帮助对照代码，不涉及 blockIdx（相当于 4×4 矩阵、2×2 block、一个 block 盖住左上角）。

### 原始 in

```text
        col:  0   1   2   3
row 0:      [ a   b   c   d ]
row 1:      [ e   f   g   h ]
row 2:      [ i   j   k   l ]
row 3:      [ m   n   o   p ]
```

### 转置 out

```text
        col:  0   1   2   3
row 0:      [ a   e   i   m ]
row 1:      [ b   f   j   n ]
row 2:      [ c   g   k   o ]
row 3:      [ d   h   l   p ]
```

规律：**out 的第 0 行 = in 的第 0 列**。

### 阶段 1：4 个线程读 in → tile

| 线程 (tx,ty) | 读 in | 写入 shared |
|--------------|-------|-------------|
| (0,0) | a | tile[0][0]=a |
| (1,0) | b | tile[0][1]=b |
| (0,1) | e | tile[1][0]=e |
| (1,1) | f | tile[1][1]=f |

对应代码：`tile[threadIdx.y][threadIdx.x] = in[...]`

### 阶段 2a：shared_simple 写 out（不合并）

| 线程 | 写 out | 值 | 线性地址 |
|------|--------|-----|----------|
| (0,0) | out[0][0] | a | 0 |
| (1,0) | out[1][0] | b | 4（跳了！） |
| (0,1) | out[0][1] | e | 1 |
| (1,1) | out[1][1] | f | 5 |

同一 warp (ty=0) 写地址 0 和 4，不连续。

### 阶段 2b：shared 写 out（合并）

| 线程 | row_out | col_out | 写 | 从 tile 取 |
|------|---------|---------|-----|------------|
| (0,0) | 0 | 0 | out[0]=a | tile[0][0] |
| (1,0) | 0 | 1 | out[1]=e | tile[1][0] |
| (0,1) | 1 | 0 | out[4]=b | tile[0][1] |
| (1,1) | 1 | 1 | out[5]=f | tile[1][1] |

warp (ty=0) 写 out[0] 和 out[1]，**连续** ✓

## 七、三个 kernel 对比

| 版本 | 函数名 | 行号 | 读 in | 写 out | 建议学习顺序 |
|------|--------|------|-------|--------|--------------|
| 1 | `transpose_naive` | 29–40 | 合并 ✓ | 不合并 ✗ | ① 理解转置下标 |
| 2a | `transpose_shared_simple` | 42–60 | 合并 ✓ | 不合并 ✗ | ② 理解 shared |
| 2b | `transpose_shared` | 62–82 | 合并 ✓ | 合并 ✓ | ③ 理解写回优化 |

### 三句话记忆

1. **Shared 是中转站**——先快读进来，再写出去。
2. **写 out 时**，让 warp 写**同一行、相邻列** → 内存连续。
3. **坐标对调了**，从 shared 取 `tile[tx][ty]` 而不是 `tile[ty][tx]`。

### 物理类比

把 shared 里 2×2 便签**转 90°**再贴回 out。  
`transpose_shared` = 转 90° + 32 人排队写一行。

### CPU 参考实现（第 88–94 行）

```cuda
void transpose_cpu(const float* in, float* out, int width, int height) {
  for (int row = 0; row < height; ++row) {
    for (int col = 0; col < width; ++col) {
      out[col * height + row] = in[row * width + col];
    }
  }
}
```

和 kernel 里 `out[col * height + row] = in[row * width + col]` 逻辑一致，用来生成正确答案。
