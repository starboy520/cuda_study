# 05 功耗、频率、散热与持续性能

## 0. 先建立大局观：标称性能 ≠ 持续性能

规格表写着"8.1 TFLOP/s"，但你的 kernel 跑久了可能达不到——因为 GPU 会因**发热**而**降频**。
本章讲清功耗、频率、散热三者的关系，解释一个反直觉现象：**同一个 kernel，跑一次很快，连跑
一小时可能变慢**。这对 benchmark（卷五）和生产部署都很关键。

```text
峰值性能（规格表）：理想散热、瞬时频率下的理论值
持续性能（实际）：  长时间运行、受功耗墙和温度墙限制的真实值
两者可能差很多 -> 测性能和部署都要考虑持续性能
```

## 0.1 术语速查表

| 术语 | 一句话定义 |
|---|---|
| **TDP** | 热设计功耗，GPU 的功耗上限（如 T4 70W）|
| **boost clock** | 加速频率，散热允许时的高频 |
| **base clock** | 基础频率，保证能持续的频率 |
| **thermal throttling** | 温度过高时自动降频保护 |
| **power throttling** | 功耗超限时降频 |
| **持续性能** | 长时间运行能维持的真实性能 |

## 1. 功耗、频率、温度的循环

三者形成一个相互制约的循环：

```text
频率高 -> 功耗大 -> 发热多 -> 温度升 -> 触发降频 -> 频率降
                                              ↓
                                         性能下降

GPU 在"想跑快(高频)"和"别过热/超功耗(降频)"之间动态平衡
```

两道"墙"限制频率：

```text
功耗墙（power limit）：功耗到 TDP 上限就降频（如 T4 的 70W）
温度墙（thermal limit）：温度过高（如 83°C+）就 thermal throttling 降频
先撞到哪道墙就被哪道限制
```

## 2. 为什么"跑久了变慢"

典型场景，解释这个反直觉现象：

```text
刚启动：GPU 凉、频率冲到 boost clock -> 性能高
跑几分钟：温度逐渐升高 -> 接近温度墙 -> 开始降频
持续运行：稳定在某个较低频率（散热能压住的频率）-> 性能比刚开始低

-> 所以"第一次跑特别快"可能是还没热起来，不代表持续性能
```

这对 **benchmark 的启示**（呼应卷五/01）：

```text
- warmup 不只是排除 JIT/context，也让频率/温度进入稳态
- 测持续性能要跑足够长，等降频稳定后再测
- 短时 burst 性能和长时持续性能是两个不同指标，要分清报告哪个
```

## 3. TDP：功耗预算决定性能上限

每块卡有 **TDP（热设计功耗）**——功耗上限：

```text
T4：     70W（低功耗，适合密集部署、被动散热）
A100：   400W（高性能，主动散热）
H100：   700W（旗舰，需强散热）
```

TDP 越高，能维持的频率/算力越高，但对散热和供电要求也越高。**T4 的 70W 是它的设计取舍**：
低功耗、能塞进密集服务器、被动散热，代价是峰值算力不如高功耗卡。

## 4. 散热方案影响持续性能

同一块 GPU，散热不同，持续性能可能差很多：

```text
数据中心服务器：强风道/液冷 -> 温度压得住 -> 维持高频 -> 持续性能好
塞在通风差的机箱：温度容易撞墙 -> 频繁降频 -> 持续性能差
被动散热卡（如 T4）：依赖服务器风道，单独裸跑会过热降频
```

> 实战意义：你在一台散热好的机器上测的性能，换台散热差的机器可能跑不出来。**报告性能要带
> 环境信息**（卷五/01、卷十/04 强调的"控制变量"包括散热/功耗环境）。

## 5. 怎么观察和考虑这些

用 `nvidia-smi` 实时看功耗、频率、温度：

```bash
nvidia-smi -q -d POWER,CLOCK,TEMPERATURE   # 详细功耗/频率/温度
nvidia-smi dmon                             # 持续监控
```

观察要点：

```text
- 跑 kernel 时功耗是否撞 TDP？(power throttling)
- 温度是否接近上限？频率有没有从 boost 掉下来？(thermal throttling)
- 长时间跑，频率是否从初始值下降并稳定？
```

对 benchmark 和部署的考虑：

```text
- 测性能：跑够长进入稳态，区分 burst vs 持续，记录环境
- 部署：按持续性能而非峰值规划容量，确保散热达标
- 密集部署（多卡一机）：注意散热互相影响，可能整体降频
```

## 6. 实践

1. 用 `nvidia-smi dmon` 监控，跑一个长时间 kernel，观察频率/温度/功耗随时间变化。
2. 对比"冷启动第一次"和"连续跑 5 分钟后"的 kernel 耗时，看有没有降频导致的变慢。
3. 查你的 GPU 的 TDP、base/boost clock。
4. 解释：为什么 benchmark 报告要注明"是否进入热稳态"？

## 7. 面试题（附参考答案）

**Q1：为什么 GPU 跑久了可能变慢？**
频率高→功耗大→发热→温度升→触发 thermal throttling 降频。刚启动 GPU 凉、跑 boost 高频很快，
持续运行温度升高后降频，稳定在散热能压住的较低频率。

**Q2：什么是 TDP？它怎么影响性能？**
热设计功耗，GPU 的功耗上限（T4 70W、H100 700W）。功耗撞 TDP 就降频。TDP 越高能维持的频率/
算力越高，但散热供电要求也越高。

**Q3：峰值性能和持续性能的区别？**
峰值是理想散热、瞬时频率下的理论值（规格表）；持续是长时间运行、受功耗墙和温度墙限制的真实
值。两者可能差很多，部署要按持续性能规划。

**Q4：这对 benchmark 有什么启示？**
warmup 要让频率/温度进入稳态；测持续性能要跑够长等降频稳定；区分 burst 和持续性能；报告要带
散热/功耗环境（控制变量）。

**Q5：同样的卡为什么在不同机器上性能不同？**
散热不同。散热好的机器温度压得住、维持高频、持续性能好；散热差的频繁 thermal throttling、
持续性能差。所以性能报告必须带环境信息。
