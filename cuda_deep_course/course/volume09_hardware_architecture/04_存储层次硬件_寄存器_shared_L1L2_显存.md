# 04 存储层次硬件：寄存器、shared、L1/L2、显存

## 0. 先建立大局观：把"软件内存空间"对应到物理硬件

卷三讲了 CUDA 的内存空间（register/shared/global...），那是**软件视角**。本章翻到**硬件视角**：
这些内存空间物理上是什么、在芯片的哪里、为什么有的快有的慢。理解了物理结构，卷三所有"为什么
shared 比 global 快"就成了显然的事实。

```text
软件内存空间          物理硬件（在哪、是什么）           速度根源
──────────────────────────────────────────────────────────────
register          SM 内的寄存器文件（SRAM）           离 ALU 最近，最快
shared / L1       SM 内的片上 SRAM                    片上，很快
L2 cache          整个 GPU 共享的片上 SRAM            片上但远些
global memory     片外显存（GDDR/HBM）               片外，最慢
```

**核心规律：离计算单元越近、越在片上，越快但越小。** 这就是存储层次金字塔。

## 0.1 术语速查表

| 硬件 | 软件对应 | 位置 | 速度 |
|---|---|---|---|
| **寄存器文件** | register | SM 内 | 最快（~1 拍）|
| **片上 SRAM** | shared / L1 | SM 内 | 快（~几十拍）|
| **L2 cache** | 缓存 global | GPU 共享 | 中（~一两百拍）|
| **GDDR / HBM** | global memory | 片外 | 慢（~几百拍）|
| **SRAM** | — | 片上静态存储，快但贵、占面积 |
| **HBM** | — | 高带宽显存（堆叠），高端卡用 |

## 1. 存储层次金字塔

GPU 的存储从快到慢、从小到大：

```text
            ┌──────────────┐
            │   寄存器     │  最快(~1拍)、最小、每线程私有
            ├──────────────┤
            │ Shared / L1  │  快(~几十拍)、小、每 SM、block 共享
            ├──────────────┤
            │  L2 cache    │  中(~百拍)、较大、全 GPU 共享
            ├──────────────┤
            │ 显存(GDDR/HBM)│ 慢(~几百拍)、大(GB级)、全局
            └──────────────┘
              离 ALU 越远越慢越大
```

为什么这样设计：**快的存储（SRAM）贵且占芯片面积，只能做小**；大容量只能用便宜的片外显存，
但慢。分层让"常用的小数据放快的、大数据放慢的"。

## 2. 寄存器文件：最快但要瓜分

寄存器是 SM 内的一块**大 SRAM**，所有驻留线程的寄存器都从这里分：

```text
- 离 ALU 最近，访问 ~1 拍，最快
- 每 SM 的寄存器文件总量固定（如 64K 个 32-bit 寄存器）
- 被所有驻留线程瓜分：每线程用得多 -> 能驻留的线程少 -> occupancy 低（卷五/03）
- 寄存器不够 -> spill 到 local memory（其实在显存，慢，卷三/01）
```

这就是卷五/03"寄存器压力影响 occupancy"的硬件根源：寄存器文件是固定大小的物理资源，瓜分它
的线程越多，每个能拿的越少。

## 3. Shared Memory 与 L1：同一块物理 SRAM

一个关键事实：**shared memory 和 L1 cache 在物理上是同一块片上 SRAM**，只是用途不同：

```text
同一块 SM 内 SRAM，分成两部分：
  shared memory：你显式管理（__shared__，卷三/03）
  L1 cache：     硬件自动缓存 global 访问

很多架构允许调整两者的划分比例（如更多 shared 或更多 L1）
```

为什么 shared 快（卷三/03）：它就是**片上 SRAM**，和 L1 一样近、一样快（~几十拍），远快于
片外显存（~几百拍）。bank 结构（卷三/03）也是这块 SRAM 的物理组织方式——32 个 bank 对应硬件
的 32 个并行访问端口。

## 4. L2 Cache：全 GPU 共享的缓冲

L2 是**整个 GPU 共享**的片上 cache，缓冲所有 SM 对显存的访问：

```text
- 比 L1/shared 大，但比显存小，比显存快
- 所有 SM 共享 -> 一个 SM 缓存的数据，另一个 SM 可能命中
- global 访问先查 L1（若启用）-> 再查 L2 -> 都未命中才去显存
```

意义（卷三/04）：

```text
- 跨步访问有时没那么慢，因为相邻 warp 重用了 L2 缓存的 cache line
  -> 但这不代表布局好，cache 容量有限、会被挤掉（卷三/04 强调过）
- 数据复用如果跨 block，要靠 L2（block 间不共享 shared）
- L2 persisting（卷三/04）能把热点数据"钉"在 L2
```

## 5. 显存：GDDR vs HBM

显存（global memory 的物理载体）是**片外**的，分两种主流类型：

```text
GDDR（如 GDDR6）：消费级/中端卡用，带宽较高（T4 约 320 GB/s）
HBM（高带宽显存）：高端卡用（A100/H100），堆叠式，带宽极高（HBM2e/3 可达 2-3 TB/s）
```

为什么显存慢（相对片上）：它**在芯片外面**，信号要走出芯片、过封装、到显存颗粒再回来，物理
距离和接口带宽限制了速度。但它**容量大**（GB 级），存得下大数据。

```text
显存的特点：容量大(GB) + 带宽高(几百GB/s~TB/s) + 但单次延迟高(几百拍)
优化策略：合并访问跑满带宽 + 数据复用减少访问次数（卷三全卷）
```

## 6. 把存储层次对应到优化策略

整卷三的优化都能用这个层次解释：

```text
"数据复用搬进 shared"（卷三/03，卷六）
  -> 把数据从慢的显存(几百拍)搬到快的片上 SRAM(几十拍)，复用多次

"合并访问"（卷三/02）
  -> 跑满显存带宽，用最少事务搬最多有用数据

"寄存器 tiling"（卷六/02）
  -> 把最热的数据放最快的寄存器(~1拍)

"L2 persisting"（卷三/04）
  -> 把热点钉在 L2，减少穿透到显存

"bank conflict + padding"（卷三/03）
  -> 优化片上 SRAM 的并行访问端口利用
```

> 一句话：**GPU 性能优化的本质，就是让数据尽量待在存储层次的高层（快的片上），减少访问底层
> （慢的显存）**。这是卷三、卷六所有技巧的统一硬件解释。

## 8. 面试题（附参考答案）

**Q1：GPU 的存储层次是怎样的？为什么这样分层？**
寄存器（SM 内，最快最小）→ shared/L1（SM 内片上 SRAM）→ L2（全 GPU 共享）→ 显存（片外，最大
最慢）。因为快的 SRAM 贵且占面积只能做小，大容量只能用慢的片外显存，分层平衡速度和容量。

**Q2：为什么 shared memory 比 global 快？**
shared 是 SM 内的片上 SRAM（~几十拍），global 在片外显存（~几百拍），物理距离和介质决定了
shared 快一个数量级。shared 和 L1 其实是同一块片上 SRAM。

**Q3：寄存器压力为什么影响 occupancy？**
寄存器文件是 SM 内固定大小的物理 SRAM，被所有驻留线程瓜分。每线程用得多，能驻留的线程就少，
occupancy 下降。用太多还会 spill 到显存。

**Q4：L2 cache 在优化中的作用？**
全 GPU 共享、缓冲显存访问。跨步访问有时没那么慢是因为相邻 warp 重用了 L2 的 cache line；跨
block 的数据复用也靠 L2（block 间不共享 shared）。但 L2 容量有限，不能依赖它掩盖坏布局。

**Q5：GDDR 和 HBM 区别？**
都是片外显存。GDDR 用于消费级/中端（T4 ~320 GB/s）；HBM 是堆叠式高带宽显存，用于高端
（A100/H100，2-3 TB/s）。HBM 带宽高得多，适合带宽敏感的大规模计算。
