# 07 计算通信重叠与 Scaling

## 0. 先建立大局观：多卡为什么常常"加卡不加速"

你把任务从 1 卡扩到 4 卡，期待快 4 倍，结果只快了 2.5 倍。为什么？因为多卡引入了**通信开销**
和**同步等待**，它们不随卡数减少，反而可能增加。本章讲两件事：怎么用**重叠**藏住通信，以及
怎么用 **scaling** 量化"扩展得好不好"。

```text
理想：K 卡 = K 倍速度
现实：K 卡 < K 倍，因为：
  - 通信开销（卡间传数据，卷八/04/06）
  - 同步等待（最慢卡拖累，卷八/05）
  - 负载不均衡
重叠和好的划分能逼近理想，但通信占比是天花板（Amdahl 又来了）
```

## 0.1 术语速查表

| 术语 | 一句话定义 |
|---|---|
| **计算通信重叠** | 一边算一边传，让通信延迟被计算藏住 |
| **strong scaling** | 固定总问题，加卡看加速 |
| **weak scaling** | 每卡问题固定，加卡同时扩大总问题 |
| **scaling 效率** | 实际加速 / 理想加速（K 卡的百分比）|
| **通信占比** | 通信时间 / 总时间，决定扩展上限 |
| **bucketing** | 把多个小通信打包成大消息 |

## 1. 计算通信重叠：藏住通信延迟

多卡的通信（如 all-reduce）如果**串行**在计算后面，就是纯开销：

```text
不重叠：[算梯度][all-reduce 等待][算梯度][all-reduce 等待]...
                ↑ 这段 GPU 在等通信，干瞪眼
```

**重叠**的思路：把通信放独立 stream，让它和**下一步的计算**并行（卷七的多 stream 思想搬到
多卡）：

```text
重叠：[算第1层梯度]
          [同步第1层梯度] [算第2层梯度]
                              [同步第2层梯度] [算第3层梯度]...
      通信和计算并行 -> 通信延迟被计算藏住 -> 接近"只有计算时间"
```

这正是分布式训练的关键优化：**反向传播算出一层梯度就立刻开始同步它，同时继续算下一层**。
PyTorch DDP 等框架自动做这个。

```text
实现要点：
  - 通信用独立 stream（不阻塞计算 stream）
  - 用 event 表达依赖（梯度算完才能同步它）
  - 小梯度打包（bucketing）成大消息，减少通信次数、提高链路利用
```

## 2. Strong Scaling：固定问题，加卡

**Strong scaling** 问：**总问题大小不变，多加卡能快多少？**

```text
1 卡算一个固定问题要 100 秒
2 卡：理想 50 秒，实际可能 55 秒（有通信）
4 卡：理想 25 秒，实际可能 32 秒
8 卡：理想 12.5 秒，实际可能 20 秒  <- 卡越多，通信占比越高，效率越低
```

strong scaling 的天花板是 **Amdahl 定律**（卷五/02）：不能并行的部分（通信、串行段）锁死了
加速上限。**卡越多，每卡分到的计算越少，但通信不减，通信占比越来越大** → 效率下降。

```text
strong scaling 效率 = (1 卡时间) / (K 卡时间 × K)
理想 = 100%，实际随 K 增大而下降
```

## 3. Weak Scaling：加卡同时加大问题

**Weak scaling** 问：**每卡的工作量固定，加卡的同时把总问题也按比例放大，时间能保持不变吗？**

```text
1 卡算规模 N，要 100 秒
2 卡算规模 2N，理想还是 100 秒（每卡仍算 N）
4 卡算规模 4N，理想还是 100 秒
-> 如果时间基本不变，weak scaling 好
```

weak scaling 更适合衡量"**用更多卡解决更大问题**"的能力（如训练更大模型）。它对通信的容忍度
比 strong scaling 高，因为每卡计算量不变、计算能持续藏住通信。

```text
strong scaling：问题不变、加卡求快   -> 受 Amdahl 限制，卡多了效率掉
weak scaling：  问题随卡变大、求时间不变 -> 更适合大规模，容忍度高
```

## 4. 怎么提升扩展效率

```text
1. 计算通信重叠（第 1 节）：藏住通信延迟，最关键
2. 减少通信量：
   - 梯度压缩 / 量化（传更少的字节）
   - bucketing 打包小消息（减少次数）
3. 用快链路：确保走 NVLink 而非 PCIe（卷八/04）
4. 负载均衡：避免最慢卡拖累（卷八/05）
5. 减少同步点：每个全局同步都是并行度损失
```

## 5. 一个量化例子

```text
任务：每卡计算 90ms + 每次 all-reduce 通信 30ms

不重叠（串行）：
  每轮 = 90 + 30 = 120ms
  4 卡 strong scaling：通信占 30/120 = 25%，效率受限

重叠（通信藏在计算后）：
  每轮 ≈ max(90, 30) = 90ms（通信被 90ms 计算藏住）
  -> 省掉 30ms，效率显著提升

结论：重叠把"计算 + 通信"变成"max(计算, 通信)"，是多卡优化的核心手段
```

## 7. 面试题（附参考答案）

**Q1：为什么多卡常常"加卡不加速"（达不到线性加速）？**
通信开销、同步等待、负载不均衡不随卡数减少。卡越多每卡计算越少但通信不减，通信占比升高，受
Amdahl 限制效率下降。

**Q2：strong scaling 和 weak scaling 区别？**
strong：固定总问题、加卡求快，受 Amdahl 限制，卡多效率掉。weak：每卡工作量固定、加卡同时放大
问题、求时间不变，更适合"用更多卡解决更大问题"，对通信容忍度更高。

**Q3：计算通信重叠是什么？怎么做？**
让通信和计算并行，藏住通信延迟。把通信放独立 stream、用 event 表达依赖（梯度算完才同步它）、
小消息打包。训练中"算下一层梯度的同时同步上一层"是典型。

**Q4：重叠对性能的本质改善是什么？**
把"计算 + 通信"（串行相加）变成"max(计算, 通信)"（并行取大）。如果通信能完全藏在计算后，
通信延迟几乎消失。

**Q5：怎么提升多卡扩展效率？**
计算通信重叠（最关键）、减少通信量（压缩/bucketing）、走 NVLink 快链路、负载均衡、减少全局
同步点。
