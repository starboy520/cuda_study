# 05 多 GPU 数据划分与同步

## 0. 先建立大局观：一块卡不够时怎么办

当数据放不下单卡显存、或单卡算太慢，就要用多 GPU。但多 GPU 不是"卡多就自动快"——怎么**切分
工作、怎么协调、通信代价多大**，决定了你能不能真正加速。本章讲多 GPU 的两种基本并行模式和
它们的同步问题。

```text
两个根本问题：
  1. 怎么把工作分给多卡？（数据并行 vs 模型/任务并行）
  2. 卡之间怎么协调？（通信、同步，代价是关键）
```

## 0.1 术语速查表

| 术语 | 一句话定义 |
|---|---|
| **数据并行** | 每卡跑完整程序，各处理一部分数据 |
| **模型/任务并行** | 把一个大问题切块，不同卡算不同块 |
| **负载均衡** | 让每卡的工作量尽量均等 |
| **`cudaSetDevice`** | 切换当前操作的 GPU |
| **集合通信** | 多卡协同的通信操作（all-reduce 等，卷八/06）|
| **scaling 效率** | K 卡实际加速 / 理想 K 倍 |

## 1. 多 GPU 编程的基本机制

CUDA 用 `cudaSetDevice` 切换"当前操作哪块 GPU"，每块卡有独立的显存和 stream：

```cpp
int numGpus;
cudaGetDeviceCount(&numGpus);

for (int g = 0; g < numGpus; ++g) {
    cudaSetDevice(g);                  // 切到 GPU g
    cudaMalloc(&d_data[g], bytes);     // 在 GPU g 上分配
    kernel<<<grid, block>>>(d_data[g]);// 在 GPU g 上启动
}
```

关键点：

```text
- 每块 GPU 有独立显存：GPU0 的指针不能直接在 GPU1 上用（除非 P2P，卷八/04）
- 每块 GPU 有独立 stream/context
- cudaSetDevice 决定后续操作落在哪块卡
- 多卡协调要么经 CPU、要么 P2P、要么用 NCCL（卷八/06）
```

## 2. 数据并行：每卡一份，各算一部分

最常用、最简单的模式。**每块卡运行相同的程序，各自处理数据的一部分**：

```text
1000 个样本，4 块 GPU：
  GPU0 算样本 0-249
  GPU1 算样本 250-499
  GPU2 算样本 500-749
  GPU3 算样本 750-999
最后汇总结果（如求和、平均）
```

典型应用：**深度学习训练**。每卡拿一部分数据算梯度，再用 all-reduce（卷八/06）把各卡梯度
求和同步——这就是分布式数据并行训练的核心。

```text
数据并行的流程：
  切分数据 -> 每卡独立计算 -> 集合通信汇总 -> 各卡继续
  通信发生在"汇总"这一步，是主要开销
```

## 3. 模型 / 任务并行：把大问题切块

当**单个问题本身太大**（如一个矩阵放不下单卡），就把问题切块分到不同卡：

```text
一个超大矩阵乘法，单卡显存放不下：
  把矩阵按行/列/块切分，每卡算一块
  块之间可能有依赖 -> 需要卡间传中间结果
```

这种模式通信更复杂（块间有数据依赖），但能突破单卡显存限制。大模型训练里的"张量并行/流水线
并行"就属于这类。

```text
数据并行：  问题不大但数据多 -> 复制程序、分数据（通信少，在汇总时）
模型并行：  问题本身太大 -> 切分问题、分块算（通信多，块间有依赖）
```

## 4. 负载均衡：别让一块卡拖后腿

多卡的总时间由**最慢的那块卡**决定。如果分配不均，快的卡算完干等慢的：

```text
不均衡：GPU0 分到 700 个，GPU1-3 各 100 个
       -> GPU0 算 7 倍时间，其余 3 卡早就闲着等 -> 浪费

均衡：  每卡约 250 个 -> 大家差不多同时完成 -> 高效
```

均衡要点：

```text
- 数据量均分（同构卡时）
- 异构卡（不同型号）按算力比例分
- 不规则负载（如 SpMV 每行不同）要动态分配
- 通信也要均衡，避免某条链路成热点
```

## 5. 同步：卡之间怎么协调

多卡协调有几种机制，代价递减：

```text
① 经 CPU 中转：GPU0 -> CPU -> GPU1，简单但慢（两段 PCIe）
② P2P 直传：   GPU 直接访问邻卡显存（卷八/04），快
③ NCCL 集合通信：all-reduce 等优化过的多卡通信原语（卷八/06），最常用
④ CUDA event 跨卡：用 event 表达卡间依赖
```

同步的核心纪律（和单卡一样，但跨卡代价更高）：

```text
- 只在必要时同步：跨卡同步比单卡 __syncthreads 贵得多
- 计算和通信重叠：边算边传，别让卡干等通信（卷八/07）
- 减少同步点：同步越多，并行度损失越大
```

## 6. 一个数据并行的骨架

把上面拼成一个多卡数据并行的流程：

```cpp
int numGpus;
cudaGetDeviceCount(&numGpus);
int chunk = n / numGpus;                          // 每卡数据量（假设整除）

// 每卡独立计算自己那部分
for (int g = 0; g < numGpus; ++g) {
    cudaSetDevice(g);
    cudaMemcpyAsync(d[g], h + g*chunk, chunk*sizeof(float),
                    cudaMemcpyHostToDevice, stream[g]);   // 分发数据
    kernel<<<grid, block, 0, stream[g]>>>(d[g], chunk);   // 各算各的
}

// 等所有卡完成
for (int g = 0; g < numGpus; ++g) {
    cudaSetDevice(g);
    cudaStreamSynchronize(stream[g]);
}

// 汇总（实际中用 NCCL all-reduce，卷八/06）
```

注意每卡用**独立 stream**，这样多卡的计算和传输能并行（不会因为共用默认 stream 串行化）。

## 8. 面试题（附参考答案）

**Q1：数据并行和模型并行的区别？**
数据并行：每卡跑完整程序、各处理一部分数据，最后汇总（数据多但问题不大，如分布式训练）。
模型并行：把一个太大的问题切块分到不同卡（问题本身放不下单卡，块间有依赖、通信更复杂）。

**Q2：多 GPU 程序的总时间由什么决定？**
由最慢的那块卡决定。所以负载均衡很关键——分配不均会让快卡干等慢卡。同构卡均分、异构卡按
算力分、不规则负载动态分。

**Q3：每块 GPU 的显存是独立的吗？怎么跨卡访问？**
是独立的，GPU0 的指针不能直接在 GPU1 用。跨卡访问要么经 CPU 中转、要么 P2P 直传（卷八/04）、
要么用 NCCL 集合通信。

**Q4：多卡为什么不是"卡越多越快"？**
通信和同步有开销，且受最慢卡限制。实际加速 < 卡数（Amdahl）——通信占比越高、负载越不均，
扩展效率越低。

**Q5：多卡程序为什么每卡要用独立 stream？**
独立 stream 让多卡的计算和传输并行；若共用默认 stream 会串行化，失去多卡并发的意义。
