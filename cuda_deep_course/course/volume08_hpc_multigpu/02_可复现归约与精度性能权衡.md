# 02 可复现归约与精度-性能权衡

## 0. 先建立大局观：为什么"每次结果都一样"很难

直觉上，同一个程序跑两次应该给出**完全一样**的结果。但 GPU 并行归约**默认做不到 bitwise
一致**——同样的输入，两次运行末位可能不同。本章讲清为什么，以及需要可复现时要付什么代价。

```text
核心矛盾：
  并行 + 浮点 = 求和顺序不固定 = 舍入误差不固定 = 结果末位会变
  而某些场景（科学计算验证、回归测试、调试）要求每次完全一致
  -> 可复现性和性能之间要权衡
```

## 0.1 术语速查表

| 术语 | 一句话定义 |
|---|---|
| **可复现 (reproducible)** | 同输入每次跑结果完全一致（含末位）|
| **bitwise 一致** | 两次结果二进制完全相同 |
| **非确定性 (non-determinism)** | 结果随运行/调度变化 |
| **atomic 归约** | 用 atomicAdd 累加，顺序由调度决定（不可复现）|
| **确定性归约** | 固定归约结构，保证顺序一致 |
| **Kahan summation** | 补偿求和，用额外变量追回舍入误差 |

## 1. 为什么并行归约不可复现

回忆卷一/卷四：GPU 上 block 的执行顺序、atomic 的到达顺序都是**调度决定的、不确定的**。而
浮点加法不满足结合律（卷八/01）。两者一结合：

```text
浮点：(a+b)+c ≠ a+(b+c)（末位可能不同）
并行：block/线程的归约顺序每次运行可能不同
→ 同样的数，不同的相加顺序，不同的舍入 → 结果末位变化
```

最典型的不可复现来源是 **atomic 归约**：

```cpp
atomicAdd(&result, partial);   // 各 block 的 partial 以"谁先到"的顺序累加
```

谁先到取决于这次的调度，所以累加顺序每次不同，`result` 末位每次可能不同。

## 2. 这是"错"吗？——通常不是

**末位的不确定性本身不是 bug**（卷四/06）。对大多数场景（图像、推理、训练）完全可以接受——
结果在容差内就对。所以第一个问题永远是：

```text
你真的需要 bitwise 可复现吗？
  不需要 -> 用最快的归约（含 atomic），末位差异容忍掉，别自找麻烦
  需要   -> 才付下面的代价（第 3 节）
```

需要可复现的典型场景：

```text
- 科学计算的结果验证（必须和参考值逐位一致）
- 回归测试（结果变了要能确定是代码改动而非随机性）
- 调试（不可复现的 bug 极难定位）
- 某些合规/审计要求
```

## 3. 怎么实现可复现归约

核心思路：**消除顺序的不确定性**，让求和顺序固定。

### 方法一：固定归约结构（不用 atomic）

```text
用确定的树形归约（卷四/03），不用 atomic：
  - 每个 block 归约自己的固定数据段，顺序固定
  - 跨 block 用多阶段 kernel（kernel 边界），而非 atomic
  - 同样的输入 -> 同样的树 -> 同样的顺序 -> 同样的结果
```

代价：比 atomic 版可能慢一点（多阶段 launch），但换来可复现。

### 方法二：提高精度，让误差小到不影响

```text
用 double 累加，或 Kahan 补偿求和：
  误差小到远低于关心的精度 -> 即使顺序变，结果在更高位都一致
```

**Kahan summation（补偿求和）**用一个额外变量追回每步丢失的低位：

```cpp
float sum = 0.0f, c = 0.0f;        // c 是"丢失的低位补偿"
for (int i = 0; i < n; ++i) {
    float y = x[i] - c;            // 补上上次丢的
    float t = sum + y;
    c = (t - sum) - y;             // 算出这次丢了多少
    sum = t;
}
```

它用几倍的计算换取接近 double 的精度，适合 FP32 大数组求和。

## 4. 精度-性能权衡：一张决策表

这是本章的核心——**不同需求选不同方案**：

| 需求 | 推荐 | 代价 |
|---|---|---|
| 只要够准、要最快 | atomic / 普通并行归约（FP32）| 不可复现、误差中等 |
| 要可复现 | 固定树形结构、避免 atomic | 略慢 |
| 要高精度 | double 累加 | 慢一半（FP64 吞吐低）、占显存多 |
| FP32 要高精度但省显存 | Kahan 补偿求和 | 几倍计算量 |
| 要极致速度、容忍低精度 | FP16 / Tensor Core | 精度低，需验证 |

> 决策口诀：**先问"需要多准、需不需要可复现"，再选方案**。别无脑上 double（慢且费显存），
> 也别无脑用 atomic（不可复现）。需求驱动。

## 5. FP64 的特别提醒：吞吐通常远低于 FP32

消费级和很多数据中心 GPU 的 **FP64 算力只有 FP32 的几分之一甚至几十分之一**：

```text
T4：FP64 吞吐极低（FP64:FP32 ≈ 1:32）
A100：FP64 较强（有专门单元），但仍低于 FP32
```

所以"为了精度全用 double"在很多卡上代价巨大。常见折中是**混合精度**：

```text
- 大部分计算用 FP32（快）
- 只在数值敏感的累加用 FP64 或 Kahan（保精度）
- 即"该快的地方快，该准的地方准"
```

## 6. 实践

1. 写一个 atomic 归约，多次运行同样输入，观察末位是否变化（验证不可复现）。
2. 改成固定树形归约（多阶段 kernel），验证多次运行结果完全一致。
3. 实现 Kahan 求和，和普通 FP32 求和对比，看它对大数组精度的改善。
4. 对比 FP32 和 FP64 归约的耗时（在你的卡上），体会 FP64 的吞吐代价。

## 7. 面试题（附参考答案）

**Q1：为什么 GPU 并行归约默认不可复现？**
block/atomic 的执行顺序由调度决定、不确定，而浮点加法不满足结合律。顺序变 → 舍入变 → 结果
末位变。atomic 归约是最典型的不可复现来源。

**Q2：末位不一致是 bug 吗？**
通常不是。多数场景（图像/推理/训练）容差内即可。只有需要 bitwise 可复现（科学验证、回归
测试、调试）时才需要处理。

**Q3：怎么实现可复现归约？**
消除顺序不确定性：用固定的树形归约结构、避免 atomic、跨 block 用多阶段 kernel；或提高精度
（double/Kahan）让误差小到不影响。代价是略慢或更费资源。

**Q4：Kahan summation 解决什么？**
用一个补偿变量追回每步舍入丢失的低位，让 FP32 求和精度接近 double，而不需要 FP64。代价是
几倍计算量，适合 FP32 大数组。

**Q5：为什么不无脑用 double 保证精度？**
很多 GPU 的 FP64 吞吐远低于 FP32（T4 约 1:32），全用 double 会大幅变慢且占双倍显存。常用
混合精度：大部分 FP32，只在敏感处用高精度。
