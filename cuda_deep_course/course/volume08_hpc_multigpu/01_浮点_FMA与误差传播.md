# 01 浮点、FMA 与误差传播

## 0. 先建立大局观：浮点不是"精确的小数"

很多 bug 源于一个错误假设：**以为计算机里的浮点数像数学里的实数一样精确**。其实浮点是
**有限位数对无限实数的近似**——大部分小数根本无法精确表示，每次运算都可能引入微小误差。

```text
0.1 + 0.2 == 0.3  ?   -> false！（0.1 和 0.2 都存不准，相加误差累积）
```

这不是 bug，是浮点的本质。GPU 上这个问题被**放大**：成千上万个数并行相加、求和顺序还和
CPU 不同，误差表现就更明显。本章讲清浮点为什么这样、误差怎么传播、什么时候该担心。

## 0.1 术语速查表

| 术语 | 一句话定义 |
|---|---|
| **IEEE-754** | 浮点数的国际标准（FP32/FP64 等的存储和运算规则）|
| **尾数 (mantissa)** | 浮点数的有效数字部分，决定精度 |
| **舍入 (rounding)** | 运算结果放不下时按规则取近似 |
| **FMA** | fused multiply-add，`a*b+c` 一步算、只舍入一次 |
| **机器精度 (epsilon)** | 1.0 之后最小可表示的增量，FP32 约 1.2e-7 |
| **灾难性消减** | 两个相近的数相减，有效位大量丢失 |
| **误差传播** | 误差随运算累积、放大的过程 |

## 1. 浮点怎么存：尾数 + 指数

FP32（单精度，32 位）的结构：

```text
[符号 1位][指数 8位][尾数 23位]
  ±        放大倍数    有效数字

值 ≈ ±1.尾数 × 2^指数
```

关键认知：**尾数只有 23 位**，能表示的有效数字有限（约 7 位十进制）。超出的部分被舍入丢弃。

```text
FP32 精度：约 7 位有效十进制数，epsilon ≈ 1.2e-7
FP64 精度：约 16 位，epsilon ≈ 2.2e-16
FP16 精度：约 3-4 位，epsilon ≈ 1e-3（深度学习用，精度低但快）
```

> 所以"差在小数点后第 7-8 位"对 FP32 是**正常的舍入**，不是 bug。判断 bug 要看差异是否超出
> 该精度的合理范围（卷四/06 的容差就是干这个）。

## 2. 为什么 `0.1 + 0.2 != 0.3`

`0.1` 在二进制里是**无限循环小数**（像十进制的 1/3），FP32 只能存它的近似值。`0.2`、`0.3`
同理。三个近似值参与运算，结果自然和精确的 `0.3` 的近似值不完全相等：

```text
0.1 -> 存成 0.100000001490...
0.2 -> 存成 0.200000002980...
相加 -> 0.300000011920...
0.3 -> 存成 0.299999997020...
两者不等 -> == 返回 false
```

这就是为什么**浮点比较永远用容差，不用 `==`**（卷四/06、卷十/03 反复强调）。

## 3. FMA：一步算 `a*b+c`，精度更高

GPU 大量使用 **FMA（fused multiply-add）**——把 `a*b+c` 作为**一条指令**计算，中间结果
`a*b` **不舍入**，只在最后舍入一次：

```text
普通两步：  t = a*b（舍入一次） -> t+c（再舍入一次）   两次舍入
FMA 一步：  a*b+c（中间不舍入，最后舍入一次）          一次舍入 -> 更准且更快
```

影响：

```text
- FMA 通常更准（少一次舍入），也更快（一条指令）
- 但它让 GPU 结果和"严格两步"的 CPU 结果可能不同 -> 又一个"差一点"的来源
- 编译器可能自动把 a*b+c 合成 FMA（卷二/06 提过），可用选项控制
```

> 面试点："为什么 GPU 和 CPU 浮点结果不同？"——除了求和顺序（卷四/06），**FMA 也是一个原因**：
> GPU 用 FMA 少舍入一次，CPU 可能没用。

## 4. 误差怎么传播：累加是重灾区

单次运算误差很小（epsilon 级），但**大量运算会累积**。最典型的是**大数组求和**：

```text
N 个数顺序相加，每步舍入误差约 epsilon，最坏情况误差随 N 线性增长
N = 100 万、FP32 epsilon ≈ 1e-7  ->  累积误差可能到 1e-1 量级！
```

这就是为什么：

```text
- reduction 的并行树形求和（卷四/03）误差反而更小：树高 log N，累积路径短
- CPU reference 要用 double 累加（卷十/03）：epsilon 小得多，基准才可信
- 大规模累加是数值稳定性的重灾区，要特别小心
```

## 5. 灾难性消减：最危险的误差放大

**两个相近的大数相减**，会让有效位大量丢失，这叫**灾难性消减（catastrophic cancellation）**：

```text
1.0000001 - 1.0000000 = 0.0000001
两个数各有 7 位有效数字，相减后结果只剩 1 位有效数字 -> 精度暴跌
```

实际场景：

```text
- 求方差：E[x²] - E[x]² 当两项相近时消减严重 -> 改用数值稳定公式
- softmax：exp 后相减大数 -> 先减最大值（卷六/04）
- 求解病态方程组 -> 误差被放大几个数量级
```

> 避免方法：重新组织计算顺序、用数值稳定的等价公式、必要时升精度。这是数值算法设计的核心
> 技巧之一。

## 6. 什么时候该担心精度

不是所有场景都要纠结浮点。判断标准：

```text
要担心：
  - 大规模累加（百万级求和、长序列）
  - 相近数相减（方差、差分）
  - 病态问题（矩阵接近奇异）
  - 需要 bitwise 可复现（科学计算验证、回归测试）
  - 用了低精度（FP16）

通常不用担心：
  - 少量运算、数值范围正常
  - 末位差异在业务容差内（图像、推理等容错场景）
```

> 一句话：**浮点误差是常态，问题是"它会不会大到影响结果"**。用容差判断，超了才查。

## 8. 面试题（附参考答案）

**Q1：为什么 `0.1 + 0.2 != 0.3`？**
0.1/0.2/0.3 在二进制里是无限循环小数，FP32 只能存近似值，近似值参与运算结果和 0.3 的近似值
不相等。所以浮点比较要用容差。

**Q2：FMA 是什么？为什么影响精度？**
fused multiply-add，把 `a*b+c` 一条指令算完、中间不舍入只最后舍一次，比分两步少一次舍入，
更准也更快。它是 GPU/CPU 结果差异的一个来源。

**Q3：为什么大数组求和误差大，怎么减小？**
顺序累加每步舍入误差累积，最坏随 N 线性增长。减小：树形归约（累积路径 log N）、用 double
累加、Kahan 补偿求和。

**Q4：什么是灾难性消减？**
两个相近的数相减导致有效位大量丢失、精度暴跌。常见于方差、差分。避免靠数值稳定的等价公式
或升精度。

**Q5：FP32 结果和 CPU 差在第 7 位是 bug 吗？**
通常不是。FP32 只有约 7 位有效数字，加上求和顺序和 FMA 差异，末位差异正常。用容差判断，超出
合理范围才是 bug。
