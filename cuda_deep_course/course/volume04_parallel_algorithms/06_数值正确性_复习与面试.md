# 06 数值正确性、复习与面试

## 1. 浮点加法不满足结合律

数学上：

```text
(a + b) + c = a + (b + c)
```

浮点中舍入可能导致两边不同。并行 reduction 改变求和树，所以结果可能与
串行顺序不同。

## 2. 验证不能只用 `==`

常见混合判断：

```text
abs(actual - expected)
<= absoluteTolerance + relativeTolerance * abs(expected)
```

容差取决于：

- 数据规模。
- 数值范围。
- 运算次数。
- 算法稳定性。
- 数据类型。

## 3. 改善精度

- 使用 double 累加。
- Pairwise/tree summation。
- Kahan 或 compensated summation。
- 缩放数据。
- 避免灾难性消减。

精度通常有性能和资源成本，应由需求决定。

## 4. 非确定性

Atomic 浮点归约中，不同运行的更新顺序可能不同，因此末位结果变化。

若业务要求 bitwise reproducibility，需要：

- 固定归约结构。
- 避免不确定顺序 atomic。
- 控制编译和硬件差异。
- 接受额外成本。

## 5.1 参考答案（先自测，再对照）

**1. `__syncthreads` 和 `__threadfence` 有何区别？**
`__syncthreads` 是 **barrier**：让一个 block 的所有线程**到齐**某点 + 保证内存可见（会让线程
等待）。`__threadfence` 是 **fence**：只保证**调用线程自己**的内存写入按顺序、对指定范围可见
（**不让任何线程等待**）。一个管"大家到齐"，一个管"我的写入别人按序看到"。

**2. Atomic 是否等于 barrier？**
不。atomic 只保证**单个地址**的读-改-写不可分割，不让线程等待、不保证大家到齐、不能原子地
同时改两个地址。barrier 管"全员到齐 + 可见性"。两者解决不同问题，不能互相替代。

**3. 不同 block 如何全局同步？**
普通 kernel 没有全 grid barrier（block 调度不确定，强行等会死锁）。可移植做法是**拆成多个
kernel**——kernel 边界就是天然全局同步墙。其他：全局 atomic 协调、cooperative launch
（`grid.sync()`，要求所有 block 同时驻留，有限制）。

**4. Warp shuffle 为什么比 shared-memory 交换更轻量？**
shuffle 让 lane 之间**直接读寄存器**，不经过 shared memory，省掉 shared 往返；且一个 warp 内
天然锁步，不需要 `__syncthreads()` 这种全 block barrier。所以 reduction 后段（≤32 个值）用
shuffle 收尾比 shared 树更快。

**5. Shuffle mask 代表什么？**
mask 声明这次 shuffle/同步**哪些 lane 参与**（32 位，每位对应一条 lane）。Volta+ 独立线程
调度下，必须如实反映"哪些 lane 还活着"，否则行为未定义。整组都在时用 `0xffffffff`。

**6. Reduction 为什么需要多阶段？**
`__syncthreads()` 只在 block 内有效，没有全 grid barrier，所以一个 kernel 只能归约到"每 block
一个部分和"。要合并跨 block 的部分和，必须靠多次 kernel launch（kernel 边界=全局同步），每轮
把规模缩小，直到剩 1 个。

**7. Scan 和 reduction 的区别是什么？**
reduction 把整个数组**压成一个值**（只要最终结果）；scan 输出**和输入等长**，每个位置是"它及
之前所有元素的归约"（前缀和）。reduction 是 scan 的"最后一个元素"。两者都靠分层并行，但 scan
要保留所有中间前缀。

**8. Histogram privatization 为什么有效？**
naive 让所有线程对 global 的 bin 做 atomic，分布集中时严重竞争、串行化。privatization 让每
block 先在 **shared 私有直方图**上 atomic（范围小、shared 快），最后每 bin 只向 global 回写
一次，把 global atomic 从"每元素一次"降到"每 block 每 bin 一次"。

**9. Stencil 为什么不能随意原地更新？**
原地更新时邻居线程可能读到"本轮已被改过"的值而非旧值，造成 race、结果不可复现。必须 ping-pong
buffer"读旧写新"，保证邻居读到一致的上一轮状态。

**10. SpMV 为什么难以实现通用最优 kernel？**
稀疏矩阵每行非零数量差异大 → 负载不均衡；`x` 间接访问不合并。thread-per-row 适合均匀短行、
warp-per-row 适合长行、load-balanced 适合极不均匀——没有一种对所有矩阵都最优，必须按行长
分布选。

**11. GPU 与 CPU 浮点结果不同是否一定是 bug？**
不一定。浮点加法不满足结合律，并行 reduction 改变了求和顺序，舍入误差不同，末位差异是**正常**
的。所以验证要用容差（`abs(a-b) <= absTol + relTol*abs(b)`），而不是 `==`。只有超出合理容差
才是 bug。

**12. 如何验证一个 reduction？**
用 **double 累加的 CPU reference** 作基准；GPU 结果与它按**容差**比较（非 `==`）；测多种规模
（含 `n=1, 31, 257, 大素数` 等边界/非整除）；配 compute-sanitizer 的 memcheck/racecheck 确认
无越界和竞争；大规模输入下再看性能。

## 7. 故障注入

1. 删除 shared reduction 循环中的 barrier。
2. 将 `start + blockDim.x < count` 删除。
3. 在 partial warp 中错误使用 full mask。
4. 使用严格 `float == doubleReference` 判断。

分别解释错误属于：

```text
race
越界
warp participation contract
数值验证错误
```
