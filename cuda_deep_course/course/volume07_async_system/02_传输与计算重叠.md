# 02 传输与计算重叠

> 这是卷七**收益最大**的一章。GPU 算得再快，如果每次都"先把数据全传过去、再算、
> 再全传回来"，PCIe 和 SM 就在轮流空等。本章用 pinned memory + 多 stream 分块，
> 把传输和计算重叠起来，并用配套 demo 实测出 2.86x 加速。

## 1. 问题：串行流水线的浪费

一个端到端任务通常是三步：

```text
H2D（拷数据到 GPU） → kernel（计算） → D2H（拷结果回 CPU）
```

朴素写法用单个 stream，三步首尾相接、严格串行：

```text
时间 →
PCIe: [==== H2D ====]                       [==== D2H ====]
SM:                   [===== kernel =====]

传的时候 SM 空着，算的时候 PCIe 空着 → 硬件一半时间在闲置
```

如果 H2D、kernel、D2H 三段时间相近，那么**理想情况下重叠能省掉约 2/3 的空闲**，
端到端逼近"只花最长一段"的时间。

## 2. 思路：分块 + 多 stream

单块数据没法重叠（它必须传完才能算）。但如果把数据**切成多块**，就能让不同块
处在不同阶段：第 2 块在传的时候，第 1 块已经在算了。

把每块丢进不同 stream（同一 stream 内 H2D→K→D2H 保持顺序，跨 stream 并发）：

```text
stream0: H2D0 ─ K0 ─ D2H0
stream1:      H2D1 ─ K1 ─ D2H1
stream2:           H2D2 ─ K2 ─ D2H2
stream3:                H2D3 ─ K3 ─ D2H3
          ↑ 任意时刻：有的块在传、有的块在算 → PCIe 和 SM 同时忙
```

这就是**软件流水线（software pipeline）**：用 stream 的并发语义，把串行的三段
错位叠起来。

## 3. 前提：必须用 pinned memory

这一步是重叠成立的**硬性前提**，漏了就前功尽弃。回忆卷三第 04 章：

- 普通 **pageable** 内存（`malloc`/`new`/`std::vector`）OS 可随时换出，GPU 的 DMA
  引擎不敢直接搬，得先偷偷拷到驱动内部的 pinned 暂存区——这次额外拷贝让
  `cudaMemcpyAsync` **退化成同步**，根本不与计算重叠。
- **pinned**（页锁定）内存（`cudaMallocHost`）OS 保证不换出，DMA 可直接搬，
  才能真正异步、和 kernel 并行。

```cpp
float* h_in;
cudaMallocHost(&h_in, bytes);   // pinned，异步重叠的前提
// ... 用完
cudaFreeHost(h_in);
```

> 一句话：**没有 pinned，就没有重叠。** 配套 demo 专门用 pageable 版做反例验证。

## 4. 核心代码（配套 lab）

`labs/07_async_system/overlap_pipeline/` 的重叠版核心：

```cpp
const int chunkSize = divUp(n, chunks);
for (int c = 0; c < chunks; ++c) {
    const int offset = c * chunkSize;
    const int cur = min(chunkSize, n - offset);          // 处理尾块非整除
    const size_t curBytes = cur * sizeof(float);
    cudaStream_t s = streams[c % nStreams];              // 轮流分配 stream

    // 同一 stream 内三步保持顺序；不同 stream 间重叠
    cudaMemcpyAsync(d_in + offset,  h_in + offset,  curBytes,
                    cudaMemcpyHostToDevice, s);
    computeKernel<<<divUp(cur, 256), 256, 0, s>>>(d_in + offset,
                                                  d_out + offset, cur);
    cudaMemcpyAsync(h_out + offset, d_out + offset, curBytes,
                    cudaMemcpyDeviceToHost, s);
}
```

几个关键点：

- 每块用 `offset` 切片，`cur` 处理尾块**非整除**情况（工业代码必须）。
- `streams[c % nStreams]`：块数可以多于 stream 数，轮流复用 stream。
- 三个异步调用都带同一个 stream `s`，保证该块内 H2D→K→D2H 有序；不同块（不同
  stream）之间由硬件并发。

## 5. 实测结果

T4 上跑 64M 元素（256 MB）、16 块、4 流：

```text
串行 (pinned, 单 stream)         99.43 ms
重叠 (pinned, 4 streams 分块)    34.77 ms   加速 2.86x
分块 (pageable, 4 streams)      100.20 ms   加速 0.99x   ← 无 pinned 不重叠
```

三个数字讲了一个完整故事：

- **2.86x**：重叠把三段串行压成近似"一段"，接近理论上限（三段相近时约 3x）。
- **pageable 0.99x**：同样写成多 stream，但 pageable 内存让 async 退化成同步，
  重叠没发生——**铁证"pinned 是前提"**。

复现：

```bash
make -C labs/07_async_system/overlap_pipeline clean all
./labs/07_async_system/overlap_pipeline/overlap_pipeline
```

## 6. 用 Nsight Systems 肉眼确认重叠

数字会说谎（可能是别的原因变快），时间线不会。用 nsys 直接看：

```bash
nsys profile --trace=cuda -o overlap_report \
  ./labs/07_async_system/overlap_pipeline/overlap_pipeline
```

在时间线上对比：

```text
串行版：  H2D 条、kernel 条、D2H 条 首尾相接，互不重叠
重叠版：  多个 stream 的拷贝条和 kernel 条 在时间轴上彼此交叠
```

看到交叠，才算真正验证了重叠发生。这是卷五第 04 章 Nsight Systems 的实战延续。

## 7. 调参与边界

- **块数 / stream 数**：不是越多越好。块太小 → 每块 launch 和同步开销占比上升；
  stream 太多 → 调度开销。常见从 `chunks=8~32`、`streams=2~4` 起步，实测调。
- **三段不均衡时**：如果 kernel 远长于传输（compute-bound），重叠收益有限，瓶颈
  本就在计算；反之传输远长于 kernel（memory-bound 传输），收益最大。先 profile。
- **双向 PCIe**：H2D 和 D2H 走不同方向，现代 GPU 有独立拷贝引擎，可让"某块在 H2D"
  和"另一块在 D2H"也并行。

## 8. 典型应用场景

```text
- 大数据集分批处理：图像/视频批量、数据量超显存需流式喂数据
- 训练数据流水线：GPU 算当前 batch，同时传下一个 batch
- 任何 "H2D + 计算 + D2H" 占比可观、且数据可切块的任务
```
