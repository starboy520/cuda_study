# 04 内存分配、复制与资源生命周期

## 1. 指针值不等于可访问性

`float*` 只描述类型，不描述内存属于 Host 还是 Device：

```cpp
float* hostPointer;
float* devicePointer;
```

两者 C++ 类型完全相同，编译器**不会**阻止你把 device 指针传给 `memcpy`、或在
Host 上写 `devicePointer[0] = 1.0f`。但运行时这会是一次非法访问，轻则段错误，
重则静默读到垃圾。根因是：CUDA 的 Host 内存和 Device 内存是**两个独立的地址空间**，
而一个裸 `float*` 不携带"我属于哪个空间"的信息。

所以命名约定（`h_`/`d_` 前缀，或 `hostPointer`/`devicePointer`）不是风格洁癖，
而是**唯一**能在编译期帮你区分二者的工具——它把类型系统丢失的信息补回到了人眼里。
本章后面所有"复制方向写错""use after free"类错误，归根到底都是这条的推论。

## 2. 字节数

```cpp
std::size_t bytes =
    static_cast<std::size_t>(count) * sizeof(float);
```

使用 `size_t` 避免大数组乘法在 `int` 中溢出。这不是杞人忧天——给个具体数字：
`int` 通常是 32 位，最大约 `2.1e9`。一个 6 亿元素的 `float` 数组需要
`600,000,000 * 4 = 2.4e9` 字节，**已经超过 `int` 上限**。如果写成
`int bytes = count * sizeof(float)`，乘法在 `int` 里先溢出成一个**负数或很小的
正数**，于是 `cudaMalloc` 只分配了一小块，后续 kernel 立刻越界。

关键在于**先转型再相乘**：`static_cast<size_t>(count) * sizeof(float)` 让整个
乘法在 64 位里进行。若写成 `static_cast<size_t>(count * sizeof(float))`，溢出
在转型**之前**就已发生，转型救不回来。

## 3. `cudaMalloc`

```cpp
float* device = nullptr;
CUDA_CHECK(cudaMalloc(&device, bytes));
```

`cudaMalloc`：

- Host 调用。
- 在当前 Device 上分配内存。
- 返回 Device pointer。
- 不初始化内容。

不要读取刚分配但未写入的数据。

## 4. 初始化

```cpp
CUDA_CHECK(cudaMemset(device, 0, bytes));
```

`cudaMemset` 按 byte 设置。设置为 0 对整数和 IEEE-754 浮点零可用，但：

```cpp
cudaMemset(device, 1, bytes);
```

不会把每个 `float` 设置为 `1.0F`。原因是 `cudaMemset` 和 C 的 `memset` 一样
**逐字节**填充：它把每个 byte 都写成 `0x01`。一个 `float` 占 4 字节，于是内存里
变成 `0x01010101`，按 IEEE-754 解释约等于 `2.4e-38`，根本不是 `1.0`。
0 之所以例外，是因为 `float(+0.0)` 的位模式恰好是 32 个 0 bit，逐字节写 0 自然
得到 0.0。要把数组初始化成非零浮点值，得用 kernel 或 `thrust::fill`，不能靠 memset。

## 5. `cudaMemcpy`

```cpp
cudaMemcpy(destination, source, bytes, direction)
```

方向包括：

```text
cudaMemcpyHostToDevice
cudaMemcpyDeviceToHost
cudaMemcpyDeviceToDevice
cudaMemcpyHostToHost
```

显式方向能暴露指针归属错误。

## 6. 生命周期

标准顺序：

```text
分配
-> 初始化/复制
-> 使用
-> 同步到安全释放点
-> 释放
```

过早释放可能与异步工作竞争；遗漏释放造成显存泄漏。

为什么"同步到安全释放点"是单独一步、不能省？因为 kernel launch 是异步的
（卷二第 05 章）。看这段**有 bug** 的代码：

```cpp
kernel<<<grid, block>>>(device);  // 异步：Host 不等它算完就往下走
cudaFree(device);                 // 危险：kernel 可能还在读 device！
```

时间线展开就一目了然：

```text
Host:   launch ──> cudaFree(device) ──> ...
GPU:         kernel 才刚开始读 device ───────X 内存已被回收
```

`cudaFree` 本身会做隐式同步，所以这个特定例子可能侥幸不崩，但一旦换成异步
`cudaFreeAsync` 或多 stream 场景，就是真正的 use-after-free。**可靠做法**是在
释放前显式插入同步点（`cudaDeviceSynchronize` 或 stream/event 同步），确保所有
读写该 buffer 的工作都已完成。`cudaMemcpy`（D2H）默认也是同步的，所以"算完→
拷回结果→再 free"这条常见顺序天然安全。

## 7. `cudaFree`

```cpp
CUDA_CHECK(cudaFree(device));
device = nullptr;
```

将本地指针置空不能撤销其他拷贝出的悬空指针，但能减少重复使用错误。

## 9. 错误场景

### 字节数写错

```cpp
cudaMalloc(&device, count); // 少乘 sizeof(float)
```

### 复制方向写错

Runtime 可能检测，也可能因统一虚拟地址环境表现不同。不要依赖自动推断掩盖
错误设计。

### Use after free

```cpp
cudaFree(device);
kernel<<<...>>>(device);
```

属于非法访问。

### 未初始化读取

使用：

```bash
compute-sanitizer --tool initcheck ./program
```

## 10. RAII 直觉

手工管理容易在异常或提前返回时泄漏。工程代码可封装：

```text
构造分配
析构释放
禁止错误复制
允许移动所有权
```
