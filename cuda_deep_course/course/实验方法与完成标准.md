# 实验方法与完成标准

CUDA 是一门必须亲手实践的技术。本页定义全书统一的实践要求。

## 1. 每章实践合同

每章至少完成五件事。

### 1.1 手写

先关闭完整答案，只根据接口、目标和输入输出要求完成代码。

可以查：

- CUDA API 签名
- 编译器错误
- 本章概念说明

不要直接复制最终 kernel。困难本身就是训练的一部分。

### 1.2 验证

GPU 结果必须与可信参考结果比较。

常见方法：

```text
CPU reference
已验证的库实现
数学不变量
小规模手工答案
```

测试至少覆盖：

```text
最小输入
小于一个 block 的输入
刚好等于一个 block 的输入
比一个 block 多一个元素
不能被 block 或 tile 整除的输入
较大性能输入
```

浮点结果根据算法使用绝对误差、相对误差或混合误差，不简单依赖 `==`。

### 1.3 故障注入

在可控输入下故意制造错误，例如：

- 删除边界判断。
- 写错行列下标。
- 删除必要同步。
- 将 `width` 和 `height` 混用。
- 制造非合并访问。
- 引入 shared-memory bank conflict。

先预测现象，再运行工具观察。完成后恢复正确代码。

### 1.4 测量

根据问题使用：

```text
CUDA Event        kernel 或 GPU 工作时间
Nsight Systems    CPU/GPU 时间线和并发
Nsight Compute    单个 kernel 的硬件指标
Compute Sanitizer 越界、竞态和未初始化访问
```

测量要求：

1. 先 warmup。
2. 重复多次。
3. 明确同步位置。
4. 不把首次上下文初始化时间误算为稳定 kernel 时间。
5. 记录 GPU、CUDA、编译参数和输入规模。
6. 报告中位数或明确说明统计方法。

### 1.5 表达

完成实验后，用自己的话回答：

```text
问题是什么？
基线实现如何工作？
瓶颈假设是什么？
改动了什么？
证据支持还是推翻了假设？
正确性和适用范围是什么？
下一步还可以怎么做？
```

这是面试表达和工程报告的共同基础。

## 2. 四级实践

### L1：跟练

跟随文档补全局部代码，并能解释每一行。

### L2：独立实现

只提供函数接口、目标、限制和验收标准。独立完成代码。

### L3：优化实验

先提出一个具体假设：

```text
我认为版本 A 慢是因为 X。
如果改成 Y，指标 Z 应该发生变化。
```

再使用工具证明或推翻。

### L4：综合项目

独立完成：

```text
需求定义
CPU reference
多个 CUDA 版本
测试
benchmark
profiling
优化报告
README
```

## 3. 正确性优先

优化顺序固定为：

```text
明确规格
  -> 写参考实现
  -> 写最简单 GPU 版本
  -> 覆盖边界测试
  -> 使用检查工具
  -> 建立性能基线
  -> 一次只验证一个优化假设
```

错误结果即使快十倍，也没有性能意义。
