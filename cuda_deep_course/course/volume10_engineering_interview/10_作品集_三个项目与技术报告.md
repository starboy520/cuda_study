# 10 作品集：三个项目与技术报告

## 0. 先建立大局观：作品集是"信任凭证"

简历上写"精通 CUDA"没人信。**一个有代码、有测试、有性能数据、有技术报告的项目**，才能让
面试官在 5 分钟内判断你的真实水平。作品集不是炫技，而是证明：

```text
你能写对（测试） + 写快（性能数据） + 讲清（技术报告） + 工程化（构建/可复现）
```

本章给三个**难度递进、覆盖面互补**的项目方案，外加一份技术报告模板。做完这三个，你就有了
面试时拿得出手的实证。

## 0.1 三个项目一览

| 项目 | 难度 | 覆盖能力 | 对应卷 |
|---|---|---|---|
| **P1. 高性能 Reduction 库** | ⭐⭐ | 归约优化阶梯、warp 原语、测试体系 | 卷四、卷十 |
| **P2. GEMM 优化与 cuBLAS 对标** | ⭐⭐⭐ | tiling、register、Roofline、profiler | 卷五、卷六 |
| **P3. 融合算子（如 Softmax/LayerNorm）** | ⭐⭐⭐ | 数值稳定、kernel 融合、端到端 | 卷六、卷七 |

> 建议：至少完成 **P2**（GEMM 是面试硬通货）。三个都做最好，时间紧就 P1 + P2。

## 1. 项目 P1：高性能 Reduction 库

### 目标

把数组归约（求和/max/点积）做成一个**可复用、有测试、有基线**的小库，并展示从 naive 到
warp-shuffle 的完整优化阶梯。

### 交付内容

```text
reduction/
├── CMakeLists.txt            # 可复现构建（卷十/01）
├── include/reduce.h          # 干净的库接口
├── src/reduce.cu             # 多版本：atomic / shared / warp-shuffle
├── tests/test_reduce.cu      # CPU reference + 多规模 + 容差（卷十/03）
├── bench/bench_reduce.cu      # 基线测量 + CSV 输出（卷十/04）
└── README.md                 # 技术报告（见第 4 节模板）
```

### 关键展示点

```text
- 优化阶梯：每版都有"动机 → 实现 → 量化收益"（卷四/03）
- 正确性：8 种规模含边界全 PASS，racecheck/memcheck 干净
- 性能：各版 GFLOPS/带宽对比表，warp-shuffle 版接近带宽墙
- profiler：ncu 证明它是 memory-bound、第一阶段逼近 DRAM 峰值
```

### 加分项

- 支持任意归约算子（模板化 sum/max/min）。
- 和 CUB `DeviceReduce` 对比，分析差距来源。

## 2. 项目 P2：GEMM 优化与 cuBLAS 对标（推荐主打）

### 目标

实现 GEMM 的完整优化阶梯（naive → tiling → register tiling），用 cuBLAS 当标尺，用 ncu 验证
每步，产出一份有说服力的优化报告。这是**最能体现 CUDA 性能工程能力**的项目。

### 交付内容

```text
gemm/
├── CMakeLists.txt
├── src/
│   ├── gemm_naive.cu         # 基线
│   ├── gemm_tiled.cu         # shared tiling
│   └── gemm_reg.cu           # register tiling
├── tests/test_gemm.cu        # 含非方阵、非整除
├── bench/bench_gemm.cu        # 四版 + cuBLAS 对比，GFLOPS 表
└── README.md                 # 技术报告
```

### 关键展示点（面试官最看重）

```text
- Roofline 定位：算 AI，说明 naive 为什么 memory-bound（卷五/02）
- 优化阶梯：tiling 提升 AI、register tiling 提升 ILP，每步量化
- 对标：手写版到了 cuBLAS 的百分之几？差距分析（向量化？双缓冲？）
- profiler 闭环：每步用 ncu 验证假设（SoL 移动、occupancy 变化）
- 3 分钟讲解稿：能口述整个优化叙事
```

### 加分项

- 加 FP16/Tensor Core 版本（若有 Ampere+ 卡）。
- 不同矩阵尺寸的性能曲线，分析何时哪个版本最优。

## 3. 项目 P3：融合算子（Softmax / LayerNorm）

### 目标

实现一个数值稳定、且做了 kernel 融合的算子，体现"算子开发"岗的核心能力。

### 交付内容

```text
softmax/
├── CMakeLists.txt
├── src/softmax.cu            # 数值稳定 + 多级归约 + 融合
├── tests/test_softmax.cu     # 和 CPU/PyTorch reference 对比
├── bench/bench_softmax.cu
└── README.md
```

### 关键展示点

```text
- 数值稳定：减最大值防溢出（softmax 的经典处理，卷六/04）
- 多级归约：一行内 max 归约 + sum 归约，warp shuffle 收尾
- 融合：max/sub/exp/sum/div 融成一个 kernel，减少 global 往返（卷六）
- 正确性：和成熟实现（PyTorch/CPU）按容差对齐
```

### 加分项

- LayerNorm/RMSNorm 变体。
- 和 PyTorch 原生算子对比性能。

## 4. 技术报告模板（每个项目都配一份）

报告是作品集的"门面"——面试官常常先看 README。用这个结构：

```markdown
# 项目名：一句话说清它是什么

## 1. 问题与目标
- 解决什么问题、性能目标是什么

## 2. 环境
- GPU 型号 / CUDA 版本 / 驱动 / 编译参数（可复现的关键）

## 3. 方法与优化阶梯
- v0 基线 → v1 → v2 ... 每步：动机 + 实现要点 + 解决什么瓶颈

## 4. 正确性验证
- CPU reference、测试规模（含边界）、容差、sanitizer 结果

## 5. 性能结果
- 各版本对比表（时间/带宽/GFLOPS），和库（cuBLAS/CUB）对标
- profiler 证据：关键指标如何支持"为什么变快"

## 6. 分析与取舍
- 瓶颈是什么、还差多少、差在哪、什么场景下结论会变

## 7. 结论与未来工作
- 学到什么、还能怎么优化
```

> 报告的灵魂是 **§3 和 §5**：每一步优化都有"动机→数据"的闭环，而不是"我加了 tiling"这种
> 无证据陈述（卷五/06 反模式）。

## 5. 作品集整体检查清单

每个项目交付前过一遍：

```text
[ ] CMake 一键构建，README 有清晰的 build/run 步骤
[ ] 所有版本正确性测试 PASS（含边界规模）
[ ] compute-sanitizer memcheck/racecheck 干净
[ ] 性能数据带完整环境信息，可复现
[ ] 至少一处 profiler（ncu/nsys）证据支持优化结论
[ ] 技术报告完整，优化阶梯每步有"动机+数据"
[ ] 代码用 RAII 管理资源、CUDA_CHECK 包错误（卷十/02）
[ ] 能口述一段 3 分钟的项目讲解
[ ] 推上 GitHub，README 是门面
```

## 6. 怎么在面试中讲项目

```text
3 分钟结构：
  10 秒  这是什么、解决什么问题
  30 秒  最简基线和它的瓶颈
  90 秒  优化阶梯：每步动机 + 量化收益（这是核心，展示你的思维）
  30 秒  最终结果 + 和库的对标 + 还差什么
  20 秒  学到什么 / 取舍

口诀：讲"问题→假设→改法→数据"的闭环，不要罗列"我用了哪些技术"。
```

3. 推上 GitHub，README 按第 4 节模板写好。
4. 时间充裕再做 P1 或 P3，凑齐 2-3 个项目。
