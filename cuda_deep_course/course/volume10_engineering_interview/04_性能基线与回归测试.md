# 04 性能基线与回归测试

## 0. 先建立大局观：没有基线，"变快了"就是错觉

你做了一个优化，跑了一下"感觉快了"——这是工程上最危险的判断。没有**基线（baseline）**和
**回归测试（regression test）**，你无法回答三个关键问题：

```text
1. 真的变快了吗？还是这次运行刚好赶上 GPU 频率高？
2. 快了多少？是 2 倍还是 2%？（决定值不值得这个复杂度）
3. 我改 A 的时候，有没有不小心把 B 改慢了？（回归）
```

**基线** = 一个被记录下来、可对比的性能参照点。**回归测试** = 自动检测"这次改动有没有让性能
变差"。两者合起来，让性能变化**有数据、可追溯、不靠感觉**。

类比：

```text
没有基线的优化  ≈ 减肥不称体重，全凭"感觉瘦了"
有基线+回归     ≈ 每天称重记录，胖了立刻发现
```

## 0.1 术语速查表

| 术语 | 一句话定义 |
|---|---|
| **baseline** | 记录下来的性能参照点（时间/带宽/GFLOPS）|
| **warmup** | 正式计时前先空跑几次，排除首次开销 |
| **回归** | 改动后性能意外变差 |
| **中位数** | 多次测量取中间值，抗异常值 |
| **P90/P99** | 90%/99% 的运行快于这个值（延迟服务关注）|
| **回归阈值** | 允许的性能波动范围（如 ±5%）|

## 1. 先回顾：怎样才算一次"可信的测量"

基线的前提是测量本身可信。卷五/01 讲过，这里收口成清单：

```text
✅ warmup：先空跑几次，排除 context 初始化、JIT、首次内存触碰（卷二/06）
✅ 多次重复：跑 N 次（如 50/100），不靠单次
✅ 取中位数：而非平均（平均易被个别异常值拉偏）
✅ 用 CUDA Event 测 GPU 时间（卷二/07），或 CPU timer + cudaDeviceSynchronize
✅ 固定环境：GPU 型号、驱动、编译参数、输入规模、Release 构建
✅ 防优化消除：结果要被读取/校验，否则编译器可能删掉"没用的"计算
```

一段标准计时骨架：

```cpp
// warmup
for (int i = 0; i < 5; ++i) kernel<<<g, b>>>(...);
CUDA_CHECK(cudaDeviceSynchronize());

// 计时 N 次
std::vector<float> times;
for (int i = 0; i < 50; ++i) {
    cudaEventRecord(start);
    kernel<<<g, b>>>(...);
    cudaEventRecord(stop);
    cudaEventSynchronize(stop);
    float ms; cudaEventElapsedTime(&ms, start, stop);
    times.push_back(ms);
}
std::sort(times.begin(), times.end());
float median = times[times.size() / 2];     // 取中位数
```

## 2. 基线该记录什么

一条基线不只是"一个时间数字"，要带上**让它可复现、可解读**的上下文：

```text
基线记录应包含：
  kernel 名 + 版本（如 reduce_v3_warpshuffle）
  输入规模（如 n=16777216）
  中位耗时（ms） + 有效带宽(GB/s) 或 GFLOPS
  GPU 型号 / 驱动 / CUDA 版本
  编译参数（-arch、-O、是否 Release）
  日期 / commit hash
```

存成机器可读的格式（CSV/JSON），方便对比和画趋势：

```csv
kernel,n,median_ms,gbps,gpu,cuda,commit,date
reduce_shared,16777216,0.82,163.8,T4,12.x,a1b2c3,2026-06-17
reduce_warpshuffle,16777216,0.45,298.4,T4,12.x,a1b2c3,2026-06-17
```

> 为什么要记 commit hash？因为半年后看到"reduce 是 0.45ms"，你要能回到**当时那份代码**才能
> 复现。性能数字脱离代码版本就没有意义。

## 3. 回归测试：自动抓"变慢了"

回归测试的逻辑很简单——**把当前性能和基线比，超过阈值就报警**：

```cpp
bool check_regression(float current_ms, float baseline_ms, float tol = 0.05f) {
    // 允许 5% 波动（GPU 频率、测量噪声）
    if (current_ms > baseline_ms * (1.0f + tol)) {
        fprintf(stderr, "PERF REGRESSION: %.3f ms vs baseline %.3f ms (+%.1f%%)\n",
                current_ms, baseline_ms,
                100.f * (current_ms - baseline_ms) / baseline_ms);
        return false;
    }
    return true;
}
```

关键是**阈值（tol）的选择**：

```text
太严（如 1%）：GPU 频率波动、测量噪声会频繁误报
太松（如 50%）：真正的性能退化被放过
经验起点：5~10%，并用中位数降低噪声
```

接进 CMake 的 `ctest`（第 01 章），就能在 CI 里**每次提交自动跑**，谁把 kernel 改慢了当场
暴露。

## 4. 正确性回归 vs 性能回归：两者都要

回归测试有两类，缺一不可：

```text
正确性回归（第 03 章）：改动后结果还对吗？     -> 容差比较，硬性 PASS/FAIL
性能回归（本章）：      改动后速度没变差吗？   -> 和基线比，超阈值报警
```

典型翻车场景：你为了修一个边界 bug 加了个 `if`，**正确性测试全过**，但那个 `if` 引入了 warp
divergence（卷五/03），kernel 慢了 30%——只有**性能回归测试**能抓到这种"修对了但改慢了"。

> 所以成熟项目的 CI 里两者并行：correctness gate（必须全过）+ perf gate（不许超基线阈值）。

## 5. 报告性能：别只报一个 speedup

汇报优化成果时，光说"快了 2 倍"信息量不足、也不可信。完整的性能报告要有**对照和上下文**
（卷五/06 反模式）：

```text
❌ "我把它优化快了 2 倍"
✅ "在 T4、n=16M、Release 下，warp-shuffle 版中位 0.45ms vs shared 版 0.82ms，
    1.82x；有效带宽从 164 到 298 GB/s（T4 峰值约 320），接近带宽墙。
    bound 类型：memory-bound（ncu SoL 验证）。"
```

差别在于后者**可复现、可质疑、可解读**——这正是面试讲优化时最该有的样子。

## 6. 实践

1. 给你的 reduction 两个版本（shared / warp-shuffle）各建一条基线（中位数 + 带宽 + 环境），
   存成 CSV。
2. 写一个 `check_regression`，故意把 kernel 改慢（如去掉 warp-shuffle 收尾），确认它报警。
3. 把性能测试接进 `ctest`，让 `ctest` 同时跑正确性和性能两类测试。
4. 写一段符合第 5 节标准的性能报告（带环境、对照、bound 类型）。

## 7. 面试题（附参考答案）

**Q1：为什么需要性能基线？**
没有基线就无法回答"真的变快了吗、快多少、有没有改慢别处"。基线是被记录、可对比的参照点，
让性能变化有数据而非靠感觉。

**Q2：一次可信的性能测量要满足什么？**
warmup 排除首次开销、多次重复取中位数、用 Event/同步正确计时、固定环境用 Release 构建、结果
被校验防编译器消除。

**Q3：为什么取中位数而不是平均？**
平均值易被个别异常值（如偶发的频率波动、调度抖动）拉偏；中位数对异常值稳健，更能代表典型
性能。

**Q4：性能回归测试怎么做？阈值怎么定？**
把当前耗时和基线比，超过阈值就报警。阈值太严会被噪声误报、太松会放过退化，经验取 5~10% 并
配中位数降噪。

**Q5：正确性测试全过，为什么还可能有性能回归？**
修 bug 加的代码可能引入 divergence、降低 occupancy、增加 shared 流量等——结果仍对但变慢。只有
独立的性能回归测试能抓到这种"修对了但改慢了"。
