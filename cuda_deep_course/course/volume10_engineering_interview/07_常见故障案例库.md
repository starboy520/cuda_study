# 07 常见故障案例库

## 0. 怎么用这个案例库

这是一本"症状 → 诊断 → 原因 → 修复"的速查手册。遇到问题时**按症状定位**，比从头读原理快。
每个案例都给出：你看到什么（症状）、怎么确认（诊断）、为什么（原因，附对应卷）、怎么修。

```text
故障分四大类：
  A. 编译/链接错误      —— 编不过
  B. 运行时崩溃/报错    —— 跑挂了
  C. 结果不对           —— 跑了但答案错
  D. 性能问题           —— 对但慢
```

> 配合工具：`compute-sanitizer`（抓越界/race）、`cuda-gdb`（断点）、`ncu`/`nsys`（性能），
> 以及卷十/02 的 `CUDA_CHECK`（让错误暴露在第一现场）。

## A 类：编译 / 链接错误

### A1. `unsupported gpu architecture 'compute_XX'`

```text
症状：nvcc 报不支持的架构
原因：你的 CUDA 版本太老，不认识这个新架构号（如旧 CUDA 编 compute_90）
诊断：nvcc --version 看版本；对照该版本支持的架构范围
修复：升级 CUDA，或改用该版本支持的架构号。（卷二/06, 卷十/05）
```

### A2. `Unresolved extern function 'xxx'`

```text
症状：链接阶段报找不到某个 __device__ 函数
原因：跨 .cu 文件调用 device 函数，但没开分离编译
诊断：确认该函数定义在另一个 .cu 里
修复：加 -rdc=true（可重定位 device 代码）。（卷二/06 §10）
```

### A3. `identifier "xxx" is undefined in device code`

```text
症状：device 代码里用了只有 host 能用的东西（如 std::cout、某些 STL）
原因：__global__/__device__ 函数里不能调用纯 host 函数
诊断：看报错的标识符是不是 host-only
修复：device 端用 printf 而非 cout；用 __host__ __device__ 标注双端可用的函数。（卷二/03）
```

## B 类：运行时崩溃 / 报错

### B1. `no kernel image is available for execution on the device`

```text
症状：程序启动 kernel 时报错
原因：编译的架构和运行的 GPU 不匹配（最常见的部署错误）
诊断：cudaGetDeviceProperties 看卡的 CC；cuobjdump --list-elf 看程序打包了哪些架构
修复：为目标卡补 -gencode，或留一份 PTX 兜底。（卷二/06, 卷十/05）
```

### B2. `an illegal memory access was encountered`

```text
症状：kernel 执行后（或下一个 CUDA 调用）报非法内存访问
原因：kernel 里越界读写（索引算错、缺边界判断、用了野指针）
诊断：compute-sanitizer --tool memcheck ./app  -> 它会指出越界的具体行
修复：补边界判断 if(idx<n)；检查索引计算。注意这是粘性错误，找第一个报错处。（卷十/02,03,06）
```

### B3. 程序卡死（hang / 永不返回）

```text
症状：kernel 启动后程序挂起
原因：divergent barrier —— __syncthreads() 放进了只有部分线程进得去的分支
诊断：compute-sanitizer --tool synccheck ./app
修复：把 barrier 移到所有线程都到得了的位置（条件只控制读写，不控制同步）。（卷四/01, 卷十/06）
```

### B4. `out of memory`

```text
症状：cudaMalloc 返回 OOM
原因：① 真的申请太多；② 显存泄漏（malloc 没 free，长跑累积）
诊断：nvidia-smi 看显存占用；检查有没有循环里反复 malloc 不 free
修复：用 RAII 管理显存（DeviceBuffer）；复用 buffer 而非反复分配。（卷十/02）
```

### B5. 错误"莫名其妙"出现在某个无辜的 CUDA 调用上

```text
症状：报错位置看着和错误无关
原因：CUDA 错误是粘性 + 异步的 —— 真正的错（如 kernel 越界）滞后到下一个调用才暴露
诊断：在每个 kernel 后加 cudaGetLastError + cudaDeviceSynchronize，定位第一个出错点
修复：用 CUDA_CHECK 包所有调用，让错误在第一现场暴露。（卷二/05, 卷十/02）
```

## C 类：结果不对

### C1. 结果每次运行都不一样

```text
症状：同样输入，多次运行结果不同
原因：race —— 多线程无保护地读写同一地址，或漏了 __syncthreads()
诊断：compute-sanitizer --tool racecheck ./app
修复：shared 读写间加 __syncthreads()；多线程更新同一地址用 atomic。（卷四/01,02, 卷十/06）
```

### C2. 结果总体对，但最后几个元素错

```text
症状：大部分元素正确，边界处错
原因：边界处理漏判 —— 非整除规模下最后一个不满的 block/tile 越界或漏算
诊断：用非整除/非方阵规模测（如 n=1003、1003x769）
修复：补 if(idx<n) 边界判断；两个加载地址要分别判边界。（卷二/08, 卷十/03,06）
```

### C3. GPU 结果和 CPU 差一点点

```text
症状：差在小数点后几位
原因：通常不是 bug —— 浮点不满足结合律，GPU 并行求和顺序不同，末位有差异
诊断：差异在容差内吗？（abs(a-b) <= absTol + relTol*abs(b)）
修复：用容差比较而非 ==；reference 用 double 累加。若需 bitwise 一致要固定归约顺序。（卷四/06, 卷十/03）
```

### C4. 转置/矩阵结果整体错乱

```text
症状：结果像是行列搞反了
原因：索引算错 —— row*width+col 写成了 col*width+row，或 width/height 传反
诊断：用非方阵（如 1003x769）测，方阵会掩盖这类 bug
修复：仔细核对行主序索引；非方阵下重新验证。（卷二/02, 卷十/06）
```

## D 类：性能问题

### D1. 优化了但没变快

```text
症状：加了 shared/tiling，时间几乎没变
原因：① 输入太小，launch 开销和噪声淹没差异；② 测的是 profiler 时间；③ 没 warmup
诊断：加大输入规模；分开测纯时间和 profiler；先 warmup 再计时
修复：用大规模 + warmup + 中位数正确测量。（卷五/01,06, 卷十/04）
```

### D2. kernel 慢，怀疑访存

```text
症状：kernel 是瓶颈，疑似 memory-bound
原因：访存不合并 / bank conflict / 无数据复用
诊断：ncu 看 Speed of Light（memory %）、gld/gst_efficiency、bank conflict 指标
修复：改合并访问、shared tile 复用、padding 消 bank conflict。（卷三/02,03, 卷五/05）
```

### D3. occupancy 低

```text
症状：ncu 显示 occupancy 低
原因：寄存器用太多 / shared 用太多 / block size 不合适
诊断：nvcc -Xptxas=-v 看寄存器和 spill；ncu 看 occupancy 限制因素
修复：减寄存器压力（精简局部变量）、减 shared、调 block size。但 occupancy 低不一定慢。（卷五/03）
```

### D4. 端到端慢，但 kernel 很快

```text
症状：kernel 计时很快，整个程序却慢
原因：瓶颈在传输/同步/CPU，不在 kernel（Amdahl）
诊断：nsys 看系统时间线 —— H2D/D2H 占比、GPU 空洞、过度同步
修复：pinned + 多 stream 重叠传输与计算；减少不必要的 cudaDeviceSynchronize。（卷五/04, 卷七）
```

### D5. 首次运行特别慢，之后正常

```text
症状：第一次启动 kernel 很慢，后续快
原因：① PTX JIT 首启编译；② context 初始化 / 首次内存触碰
诊断：是不是只打包了 PTX 没打 cubin？
修复：性能测试先 warmup；部署时为目标卡打 cubin 避免 JIT。（卷二/06, 卷五/01, 卷十/05）
```

## E. 通用排查流程

遇到不在上面的问题，按这个顺序：

```text
1. 让错误暴露：所有 CUDA 调用包 CUDA_CHECK，kernel 后查 last error + sync
2. 分类：编不过(A) / 跑挂(B) / 结果错(C) / 慢(D)
3. 上工具：
   越界/race      -> compute-sanitizer (memcheck/racecheck/synccheck/initcheck)
   性能           -> ncu(单 kernel) / nsys(系统时间线)
   断点调试       -> cuda-gdb
4. 最小复现：缩小到能稳定触发的最小输入
5. 对照原理：回到对应卷确认假设
```

## 实践

1. 故意制造 B2（越界）、C1（race）、C2（边界漏判）各一个，用对应 sanitizer 工具抓出来。
2. 把一个只打包 PTX 的程序在新容器里跑，观察 D5 首启 JIT 慢。
3. 给自己建一个个人故障库：每次踩坑记一条"症状→原因→修复"。
