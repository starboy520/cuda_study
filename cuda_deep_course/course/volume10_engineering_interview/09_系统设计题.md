# 09 系统设计题

## 0. 先建立大局观：系统设计题考什么

到了高级/资深岗，面试不只问"这个 kernel 怎么写快"，还会问"**给你一个端到端的需求，你怎么
设计整个 GPU 方案**"。这类题没有唯一答案，考的是：

```text
- 会不会先问清约束（数据量、延迟/吞吐目标、硬件、精度）
- 能不能估算（显存够不够、带宽够不够、要几块卡）
- 知不知道瓶颈在哪（传输？计算？同步？）
- 能不能权衡取舍（速度 vs 显存 vs 复杂度）
```

**答题套路（通用框架）**——任何系统设计题都按这五步走，比直接给方案更能展示能力：

```text
1. 澄清约束：数据规模？延迟还是吞吐优先？什么 GPU？精度要求？
2. 粗估算：  显存需求、计算量、传输量，先判断瓶颈在哪
3. 给基线：  最简单能跑的方案
4. 优化迭代：针对瓶颈逐步改进，每步说清解决什么
5. 谈取舍：  方案的代价、什么场景下会变
```

## 0.1 估算用的关键数字（T4 为例，记住量级）

| 量 | T4 近似值 | 用途 |
|---|---|---|
| FP32 算力 | 8.1 TFLOP/s | 估计算时间 |
| 显存带宽 | 320 GB/s | 估访存时间 |
| 显存容量 | 16 GB | 估能放多大数据 |
| PCIe 带宽 | ~16 GB/s（Gen3 x16）| 估 H2D/D2H 传输时间 |
| Roofline 拐点 | ~25 FLOP/byte | 判断 memory/compute bound |

> 估算口诀：**算时间 = FLOP / 算力，访存时间 = 字节 / 带宽，传输时间 = 字节 / PCIe**。三者
> 取最大就是瓶颈。

## 1. 题型一：显存预算（放不放得下）

### 例题："要在 T4（16GB）上跑一个 N×N 的 FP32 矩阵运算，N 最大能多大？"

```text
分析：
  一个 N×N FP32 矩阵 = N² × 4 字节
  GEMM 通常要 A、B、C 三个矩阵 = 3 × N² × 4
  令 3 × N² × 4 <= 16 GB
  N² <= 16e9 / 12 ≈ 1.33e9
  N <= 36500 左右

结论：N 约 3.6 万是上限，还要留余量给中间 buffer/对齐/其他分配，
      实际取 3 万更稳。超过就要分块（tiling）或多卡。
```

**追问应对**："如果 N 要到 10 万怎么办？" → 单卡放不下，分块到显存能容纳的子块逐块算，或
多 GPU 切分（题型四）。

## 2. 题型二：延迟 vs 吞吐（优化目标不同，方案不同）

### 例题："设计一个推理服务的 GPU 调度，怎么权衡延迟和吞吐？"

```text
先澄清：是在线服务（延迟优先）还是离线批处理（吞吐优先）？两者方案相反。

吞吐优先（离线/批量）：
  - 大 batch 喂满 GPU，提高利用率
  - 多 stream 重叠传输与计算
  - 可以接受单请求延迟高一点

延迟优先（在线/实时）：
  - 小 batch 甚至 batch=1，减少单请求等待
  - 关注 P99 延迟而非平均
  - 可能牺牲 GPU 利用率换响应速度
  - 用 CUDA Graph 消除重复 launch 开销（卷七）

折中：动态 batching —— 攒一小段时间的请求凑成 batch，在延迟上限内最大化吞吐
```

> 关键认知：**延迟和吞吐常常对立**。大 batch 利用率高（吞吐好）但单请求要等凑批（延迟差）。
> 设计前必须先问清楚优化哪个。

## 3. 题型三：传输瓶颈（端到端优化）

### 例题："一个流水线 H2D→kernel→D2H 反复执行，怎么优化？"

```text
1. 估算各段：假设 H2D 3ms、kernel 1ms、D2H 3ms，串行一轮 7ms，计算只占 14%
2. 判断瓶颈：传输（PCIe）是大头，不是计算
3. 优化：pinned memory + 多 stream 分块流水线，让传输和计算重叠（卷七/02）
        块 i+1 的 H2D 与块 i 的 kernel、块 i-1 的 D2H 并行
4. 理想结果：总时间从 3+1+3 向 max(3,1,3) 靠拢
5. 进一步：减少传输总量（数据能否留在 GPU？）、用 CUDA Graph 减 launch 开销
```
（卷五/04、卷七）

## 4. 题型四：多 GPU 切分（单卡不够时）

### 例题："数据/模型单卡放不下或算太慢，怎么用多卡？"

```text
先分清两种并行：
  数据并行：每卡一份完整程序，各算一部分数据，最后汇总（如各算一批样本）
  模型/任务并行：把一个大问题切块分到不同卡（如大矩阵按块分）

通信是关键代价：
  卡间通信走 NVLink（快）或 PCIe（慢），通信量大时成瓶颈
  用 NCCL 做 all-reduce / broadcast 等集合通信
  计算与通信尽量重叠（边算边传）

估算：单卡算 T 秒，理想 K 卡 T/K，但要加通信开销和负载不均衡损失
     -> 实际加速 < K，要看通信占比（Amdahl again）
```
（卷八）

## 5. 题型五：精度 vs 性能

### 例题："这个算子能不能用 FP16/混合精度加速？"

```text
权衡：
  FP16/Tensor Core 算力远高于 FP32，访存也减半 -> 快
  但精度降低，可能影响数值稳定性（如大规模累加、softmax）

设计：
  - 算力密集且容忍精度的部分用 FP16（如 GEMM 主体）
  - 数值敏感的累加用 FP32（混合精度：FP16 算、FP32 累加）
  - 验证：和 FP32 reference 比，误差在业务容差内吗？（卷四/06）

结论：不是无脑降精度，要按算子的数值敏感度分别决定
```

## 6. 一个完整的系统设计回答示例

**题："设计一个把大图像做卷积的 GPU 方案。"**

```text
1. 澄清：图多大？filter 多大？实时还是批处理？什么 GPU？精度？
   假设：4K 图像、5x5 filter、批处理吞吐优先、T4、FP32

2. 估算：4K 图 ≈ 3840×2160×4B ≈ 33MB，远小于 16GB，单卡放得下
   卷积计算量大，可能 compute-bound 或 memory-bound（取决 filter 大小）

3. 基线：naive 卷积，每输出读 5x5 窗口

4. 优化：
   - shared tile + halo 复用重叠窗口（卷四/05）
   - filter 放 constant memory（小、只读、广播）
   - 多图用多 stream 重叠传输与计算
   - 用 ncu 验证瓶颈，针对性优化

5. 取舍：tile 大小权衡复用 vs occupancy；要不要上 FP16 看精度需求
```

1. 给自己出题：估算"在 T4 上做 8192×8192 FP32 GEMM"的显存需求和理论计算时间。
2. 用第 0 节五步框架，口头回答"设计一个 reduction 服务处理持续到来的数据流"。

## 8. 面试技巧（怎么答好系统设计题）

```text
✅ 先问约束再答 —— 不澄清就给方案是大忌
✅ 边估算边说 —— 展示你能定量判断瓶颈
✅ 从简单基线开始，逐步优化 —— 不要一上来就上最复杂方案
✅ 主动谈取舍 —— 说清每个选择的代价，体现工程成熟度
✅ 承认不确定 —— "这取决于实测，我会用 ncu 验证"比硬猜好
❌ 别背模板 —— 面试官会追问，背的撑不过三个 why
```
