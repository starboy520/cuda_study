# 04 延迟隐藏与大量线程

## 1. 问题：访问 Global Memory 时怎么办

当一个 warp 发出 global memory 请求后，数据不会立刻到达。如果没有其他
工作可做，执行资源就会等待。

CPU 常依靠：

- 大缓存。
- 乱序执行。
- 推测执行。
- 少量硬件线程。

GPU 的重要方法是：

> 保持多个可运行 warp。当一个 warp 等待时，调度另一个已经准备好的 warp。

## 2. 一个简化时间线

假设 SM 上只有一个 warp：

```text
Warp A: 发出加载 ----------等待---------- 计算
SM:     工作               空闲            工作
```

如果有多个可运行 warp：

```text
Warp A: 加载 ----------等待---------- 计算
Warp B:       加载 ----------等待---------- 计算
Warp C:             计算  加载 ----------等待
SM:     A工作 B工作 C工作 其他warp工作 ...
```

等待并没有消失，但可以被其他 warp 的工作覆盖。这叫 latency hiding。

### 2.1 Active、Ready 与 Stalled

```text
Active warp：状态驻留在 SM 上
Ready warp：当前下一条指令已经具备执行条件
Stalled warp：等待数据、依赖、barrier 或管线
```

Occupancy 高表示 active warp 多，不保证 ready warp 多。

## 3. 为什么 CUDA 程序常启动很多线程

如果 GPU 有很多 SM，而程序只启动一个小 block：

- 大量 SM 没有工作。
- 一个 SM 上可选择的 warp 很少。
- 很难隐藏长延迟。

通常需要足够多的 block 和 warp，让硬件调度器有工作可分配。

但“线程越多越快”也不成立。过多 thread 只是排队，真正重要的是：

- 是否有足够并行工作占用 GPU。
- 每个 block 消耗多少寄存器和 shared memory。
- warp 是否经常等待数据。
- 指令和访存是否高效。

### 3.1 Grid 并行和 SM 内并行

需要两个层次：

```text
足够多 block -> 让所有 SM 有工作
每个 SM 足够多 warp -> 隐藏 SM 内等待
```

只有一个超大 block 不能跨 40 个 SM；很多很小 block 又可能每个 block 协作
效率低。

## 4. Occupancy 的第一层认识

Occupancy 常表示：

```text
实际驻留 active warp
----------------------
硬件允许的最大 active warp
```

高 occupancy 可能提供更多 warp 来隐藏延迟，但它不是最终目标。

两个重要结论：

```text
occupancy 太低可能暴露延迟
occupancy 更高不保证 kernel 更快
```

因为性能还受：

- 内存带宽。
- 指令依赖。
- 数据复用。
- cache 命中。
- 分支分歧。
- 算术吞吐。

影响。

## 5. 资源为何限制驻留 Warp

一个 SM 的资源有限，包括：

- 最大 resident thread/warp/block 数。
- Register file。
- Shared memory。

如果一个 block 使用大量 shared memory，SM 可能只能同时容纳较少 block。
如果每个 thread 使用很多寄存器，同样会减少可驻留 thread。

这就是为什么优化一个局部值可能影响整个 SM 的并发能力。

### 5.1 T4 Shared Memory 算术

T4 实测 shared memory per SM 为 64 KiB。

若每 block 使用 32 KiB，仅从 shared memory 看：

```text
最多 2 block/SM
```

若每 block 使用 48 KiB：

```text
最多 1 block/SM
```

还要同时满足 thread、register、block 数上限。

### 5.2 Thread 算术

T4 每 SM 最大 1024 resident thread：

```text
block=256 threads -> 线程上限允许 4 block/SM
block=512 threads -> 线程上限允许 2 block/SM
block=1024 threads -> 线程上限允许 1 block/SM
```

这是单因素上限，不是最终 occupancy。

## 6. Warp 需要“可运行”

SM 上存在很多 warp，不代表它们都可以立刻执行。

Warp 可能因为以下原因等待：

- Global memory 数据未返回。
- 前一条指令结果尚未准备好。
- Barrier 同步。
- 执行管线繁忙。

调度器需要从“已经准备好”的 warp 中选择。

### 6.1 Instruction-Level Parallelism

单个 thread 若有彼此独立的计算，硬件可能在等待一个结果时推进其他指令。
性能来自：

```text
Thread-level parallelism
+ Instruction-level parallelism
+ Memory-level parallelism
```

不是只靠提高 thread 数。

## 7. 一个常见误解

错误说法：

> GPU 在一个线程等待时保存它的上下文，然后像操作系统线程一样切换。

更好的初学理解：

> 多个 warp 的执行状态已经常驻硬件资源中。调度器从已经准备好的 warp 中
> 选择一个，让它的下一条指令进入执行管线。这个动作常称为“发射指令”。
> 它不是 CPU 操作系统那种昂贵的软件上下文切换。

## 9. 本章小结

```text
GPU 用多个可运行 warp 覆盖等待时间。
大量线程的目的之一是提供足够的并行和调度选择。
Occupancy 是手段，不是性能目标。
资源使用、访存和指令行为共同决定速度。
```
