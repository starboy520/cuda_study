## 4. 字段解释

### GPU name

确定具体设备型号。不同 T4 机器可能由于虚拟化或容器环境展示略有差异。

### Compute capability

表示 CUDA 架构能力版本。例如 T4 是 7.5，编译实际硬件代码时常见：

```bash
nvcc -arch=sm_75 ...
```

Compute Capability 不是 CUDA Toolkit 版本。

### SM count

GPU 中 SM 数量。它不等于 CUDA Core 数，也不等于最多 block 数。

### Warp size

当前 CUDA GPU 为 32。不要把“warp 32”推导成“block 必须只有 32 thread”。

### Max threads per block

单个 block 的 thread 总数上限。二维和三维 block 还必须满足各维度上限。

### Max threads per SM

一个 SM 可驻留 thread 数的架构上限。实际驻留数量还受寄存器、shared memory、
block 上限等约束。

### Registers

程序输出 block 和 SM 级的寄存器资源上限。源代码中的局部变量数量不能直接
等同于最终寄存器数量，应查看编译器报告或 profiler。

### Shared memory

`per block` 限制一个 block 的 shared-memory 用量；`per SM` 影响多个驻留
block 如何共享 SM 资源。

### L2 cache

L2 是整个 GPU 上的重要缓存层。卷三和卷九会深入。

### Memory clock 和 bus width

程序根据二者给出近似规格带宽：

```text
2 x memory clock x bus width / 8
```

这里的 `2` 来自双倍数据率。该结果是规格估算，不是 kernel 实测带宽。

### Async engine count

表示设备的异步复制引擎能力信息之一。是否真正实现拷贝与计算重叠，还取决于
pinned memory、stream、方向和系统条件。

## 4.1 用输出做资源算术

当前 T4 输出：

```text
SM count = 40
warp size = 32
max threads/SM = 1024
registers/SM = 65536
shared memory/SM = 64 KiB
```

### 例一：256-thread block

线程限制：

```text
1024 / 256 = 4 block/SM
```

若每 thread 使用 64 register：

```text
每 block register = 256*64 = 16384
65536/16384 = 4 block/SM
```

若每 block 再使用 32 KiB shared：

```text
64/32 = 2 block/SM
```

最终三者取最严格值，最多 2 block/SM，还需考虑架构 block/warp 上限。

### 例二：寄存器压力

若每 thread 使用 100 register：

```text
256*100 = 25600 register/block
65536/25600 = 2 block（向下取整）
```

实际分配还有粒度规则，最终应使用 occupancy API 或 Nsight Compute。

