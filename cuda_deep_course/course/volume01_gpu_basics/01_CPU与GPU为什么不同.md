# 01 CPU 与 GPU 为什么不同

## 1. 本章解决什么问题

为什么同样是处理器，CPU 常常只有几十个强核心，而 GPU 却使用大量较简单
的执行资源？为什么不能把所有程序都扔到 GPU 上？

## 2. 先看一个任务

假设要把一千万个浮点数分别加一：

```cpp
for (int i = 0; i < count; ++i) {
  output[i] = input[i] + 1.0F;
}
```

每个元素的工作彼此独立。只要输入准备好，`output[10]` 不需要等待
`output[9]`。这种任务拥有大量数据并行性。

再看另一个任务：

```text
读取配置
判断多种状态
访问几个不规则的数据结构
等待网络或磁盘
根据前一步结果决定下一步
```

它包含复杂控制流和串行依赖，不能简单拆成百万份相同工作。

GPU 更适合第一类，CPU 更擅长第二类。

## 3. CPU 的核心目标：低延迟

CPU 希望让一个或少量执行线程尽快完成。典型手段包括：

- 强大的分支预测。
- 乱序执行。
- 大容量、低延迟缓存。
- 复杂的单线程调度逻辑。
- 较高的单核性能。

可以把 CPU 想成少量经验丰富、工具齐全的工程师。单个任务遇到意外时，
他们有很多办法快速处理。

### 3.1 芯片资源花在哪里

同样有限的晶体管、面积和功耗预算，可以用于不同目标。CPU 往往投入更多资源
帮助少量指令流快速前进：

```text
复杂分支预测
大容量 cache
乱序执行窗口
重命名与调度逻辑
低延迟执行路径
```

当程序出现 cache miss 或前后依赖时，CPU 尝试从同一线程附近找出其他可执行
指令。这叫 instruction-level parallelism。

### 3.2 CPU 核心少不等于 CPU 不并行

CPU 同样拥有：

- 多核心。
- SMT/Hyper-Threading。
- SIMD/向量指令。
- 多级 cache 和预取。

“CPU 串行、GPU 并行”是错误二分法。更准确的是两者对并行和延迟的资源配置
不同。

## 4. GPU 的核心目标：高吞吐

GPU 更关心单位时间完成多少相似工作，而不是某一个线程最快多久结束。

典型手段包括：

- 提供大量并行执行资源。
- 将线程组成 warp，以相同指令批量推进。
- 使用大量可运行 warp 隐藏等待。
- 将更多芯片面积用于算术和数据吞吐。

可以把 GPU 想成许多工作组。单个工人不一定比 CPU 核心强，但当任务能拆成
大量相似小工作时，总吞吐非常高。

### 4.1 GPU 牺牲了什么

GPU 不等于“把很多 CPU 核塞在一起”。为了提高总吞吐，它更依赖：

```text
大量数据并行
较规则控制流
较规则数据访问
许多可运行 warp 隐藏等待
```

如果工作高度串行或每个线程走完全不同的路径，大量执行资源可能闲置。

### 4.2 三种并行性

```text
Task parallelism:
  不同任务同时执行，例如 CPU 做预处理、GPU 做计算。

Data parallelism:
  对大量数据执行相同操作，例如 vector add。

Pipeline parallelism:
  不同阶段处理不同批次，例如 H2D、kernel、D2H 重叠。
```

CUDA kernel 最直接表达数据并行，但完整系统会组合三种并行性。

## 5. 延迟和吞吐不是一回事

```text
延迟：一件工作从开始到结束需要多久
吞吐：单位时间一共完成多少件工作
```

例如，一辆跑车送一个包裹可能更快；一列货运列车单个包裹到达得未必更早，
但一次能运输更多包裹。

CUDA 优化经常追求吞吐，但实时系统也可能关心延迟。必须先知道业务目标。

### 5.1 数字例子

设备 A 每项延迟 1 微秒，一次只能处理 1 项：

```text
吞吐约 1,000,000 项/秒
```

设备 B 每批延迟 100 微秒，但一次处理 10,000 项：

```text
吞吐约 100,000,000 项/秒
```

B 单项结果等待更久，却拥有更高批量吞吐。GPU workload 常通过 batching 获得
这种收益。

### 5.2 Throughput 也有层次

```text
Kernel throughput
GPU device throughput
端到端 request throughput
系统多用户 throughput
```

只提高 kernel throughput，不一定提高端到端 throughput。

## 6. GPU 加速包含额外成本

一个最普通的 CUDA 程序通常需要：

```text
CPU 准备数据
  -> 将数据复制到 GPU
  -> 启动 kernel
  -> GPU 计算
  -> 将结果复制回 CPU
```

如果计算量很小，数据复制和启动开销可能比计算本身还大。

所以：

```text
GPU 计算更快
```

不自动等于：

```text
整个应用端到端更快
```

### 6.1 一个简单成本模型

```text
T_gpu_total =
    T_prepare
  + T_H2D
  + T_launch
  + T_kernel
  + T_D2H
  + T_sync
```

只有当：

```text
T_gpu_total < T_cpu
```

才是端到端加速。

### 6.2 小任务为何常不划算

假设 CPU 计算 20 微秒，GPU kernel 只需 2 微秒，但传输和 launch 总共
30 微秒：

```text
GPU 总时间 = 32 微秒
```

GPU kernel 快 10 倍，应用却更慢。

可行改进：

- 批处理更多元素。
- 将多个操作留在 GPU 上连续执行。
- 避免来回复制。
- 融合小 kernel。
- 使用异步流水线。

## 7. 哪些问题通常适合 GPU

- 大规模向量和矩阵运算。
- 图像、视频和信号处理。
- 规则网格上的 stencil。
- 深度学习训练和推理。
- 大量独立粒子或样本的模拟。
- 可批量处理的数值计算。

常见特征：

```text
数据规模大
并行工作多
每份工作相似
数据访问可以组织
CPU-GPU 传输成本能够摊薄
```

### 7.1 问题诊断表

| 问题 | 越偏“是”越适合 GPU |
|---|---|
| 能产生远多于 SM 数的并行工作吗？ | 是 |
| 每份工作执行相似指令吗？ | 是 |
| 数据访问连续或可分块复用吗？ | 是 |
| 每次传输后能做足够多计算吗？ | 是 |
| 能批处理吗？ | 是 |
| 允许一定浮点顺序变化吗？ | 是 |

## 8. 哪些问题通常不适合直接搬到 GPU

- 数据规模很小。
- 强串行依赖。
- 大量不可预测分支。
- 不规则、随机且缺乏局部性的内存访问。
- GPU 计算很少，但 CPU-GPU 往返频繁。

这不是绝对结论。有些不规则算法可以重新设计，但需要更多工作。

## 9. Amdahl 定律的直觉

假设程序中：

```text
90% 可以被 GPU 加速
10% 必须串行执行
```

即使把可加速部分变得无限快，总时间仍保留原来的 10%，理论总加速比不会
超过 10 倍。

这提醒我们：

> 优化单个 kernel 前，先确认它是不是端到端瓶颈。

卷五会正式学习 profiling、Amdahl 定律和 APOD 方法。

### 9.1 有限加速数字例子

程序原来 100 ms：

```text
20 ms 串行
80 ms 可并行
```

若 GPU 将并行部分加速 20 倍：

```text
新时间 = 20 + 80/20 = 24 ms
总加速 = 100/24 = 4.17 倍
```

不是 20 倍。继续把 GPU 部分加速到无限快：

```text
极限时间 = 20 ms
极限加速 = 5 倍
```

因此优化后瓶颈会移动。

## 9.2 算术强度的预告

GPU 有很强计算吞吐，但计算需要数据。一个算法若每搬运很多 bytes 只做一次
运算，常受内存带宽限制；若数据被重复使用并执行大量运算，更可能利用计算
单元。

卷五会用：

```text
arithmetic intensity = FLOP / byte
```

正式分析。

## 11. 本章小结

```text
CPU 偏向低延迟和复杂控制。
GPU 偏向大量相似工作的总吞吐。
GPU 加速取决于并行性、数据移动和端到端瓶颈。
```
