# 03 内存层次第一印象

## 1. 为什么需要多级内存

我们希望内存同时具备：

```text
容量大
速度快
离执行单元近
成本低
```

现实中无法全部满足，因此 GPU 使用多级存储。越靠近执行单元，通常越快、
越小、越受资源限制。

本章只建立 Register、Shared Memory、Global Memory 的第一层模型。

### 1.1 延迟与带宽

```text
Latency：一次访问等待多久
Bandwidth：单位时间能搬多少数据
```

Global memory 可以同时具有高延迟和高带宽：单个请求返回慢，但大量并行、连续
请求可以让总数据流很大。

### 1.2 数据移动的层次

```text
Host RAM
  <-> PCIe/NVLink
Device DRAM
  <-> L2/L1
SM 内 shared/register
  <-> execution pipelines
```

越靠近算术管线通常容量越小，管理方式也不同。

## 2. Register

Register 是每个 thread 保存局部值的主要位置。

```cpp
const int index = blockIdx.x * blockDim.x + threadIdx.x;
float value = input[index];
```

`index`、`value` 可能由编译器放入寄存器。

第一层特点：

```text
归属：单个 thread
共享：不能被其他 thread 直接读取
生命周期：通常随 thread
速度：非常快
容量：有限
```

寄存器使用太多会限制一个 SM 同时驻留多少 warp。这个问题在卷五和卷九深入。

### 2.1 源代码变量不等于一个寄存器

编译器会：

- 消除变量。
- 复用寄存器。
- 将值保存在常量或立即数。
- 因压力将值 spill 到 local memory。

使用：

```bash
nvcc -Xptxas=-v ...
```

观察最终资源。

## 3. Shared Memory

Shared memory 是一个 block 内所有 thread 可共享的片上存储：

```cpp
__shared__ float tile[32][33];
```

第一层特点：

```text
归属：一个 block
共享：同一 block 的 thread
生命周期：随 block
速度：通常远快于访问 DRAM
容量：较小，并消耗 SM 资源
```

典型用途：

- 让多个 thread 复用从 global memory 读入的数据。
- 在 block 内交换数据。
- 实现 tiled matrix multiply、transpose、reduction。

Shared memory 需要正确同步：

```cpp
tile[threadIdx.x] = input[index];
__syncthreads();
// 此后再读取其他线程写入的 tile 元素
```

### 3.1 生命周期例子

Block A 和 Block B 各有自己的 `tile`：

```text
Block A tile != Block B tile
```

Block A 完成后，它的 shared memory 内容消失，资源可供后续 block 使用。

## 4. Global Memory

Global memory 是 GPU 上容量最大的主要设备内存。`cudaMalloc` 通常在这里
分配：

```cpp
float* deviceData = nullptr;
cudaMalloc(&deviceData, bytes);
```

第一层特点：

```text
范围：整个 device 可访问
容量：大
延迟：高
带宽：高，但必须组织好访问
生命周期：由分配和释放决定
```

“延迟高”和“带宽高”可以同时成立：

- 单次访问等待时间可能很长。
- 大量连续访问可以获得很高总吞吐。

### 4.1 为什么 Warp 地址重要

若一个 warp 的 32 个 thread 读取连续 float（这里 lane 只是 warp 内 thread
编号）：

```text
地址：0,4,8,...,124
```

硬件可用少量事务覆盖。若地址相隔很远：

```text
0,128,256,...
```

真正请求仍只有 128 bytes，却可能触发大量事务。

## 5. 三者对比

| 存储 | 主要归属 | 可共享范围 | 容量直觉 | 常见用途 |
|---|---|---|---|---|
| Register | Thread | 单线程 | 最小 | 局部变量、累加器 |
| Shared | Block | block 内线程 | 小 | 数据复用、线程协作 |
| Global | Device | 所有 kernel | 大 | 输入、输出、主要数据 |

这张表是第一层模型，不代表全部内存系统。后续还会学习：

- Local memory
- Constant memory
- Texture/read-only 路径
- L1/L2 cache
- Pinned host memory
- Unified Memory

## 6. 数据移动常常比计算更贵

向量加法每个元素只做一次加法，却需要：

```text
读 a
读 b
写 c
```

对 `float` 而言，约移动 12 bytes，只进行 1 次浮点加法。这类 kernel 很可能
受内存带宽限制，而不是算术单元限制。

矩阵乘法如果通过 tile 重用数据，就能用一次读取支持多次乘加，算术强度更高。

### 6.1 对比

Vector add：

```text
读 A 4B + 读 B 4B + 写 C 4B
做 1 次加法
```

Naive GEMM 的一个输出：

```text
循环 K 次，每次乘加
```

虽然 naive 仍重复访存，但计算/输出比例明显更高。

## 7. Shared Memory 不是自动加速

如果只是：

```text
global -> shared -> global
```

却没有复用、重排或协作价值，那么增加 shared memory 可能只会增加指令和同步。

使用它前先回答：

1. 哪些 global 数据会被重复使用？
2. 哪些线程需要交换数据？
3. 能否改善 global memory 的访问方式？
4. 同步成本是否值得？

## 8. 实践

为下面变量判断它最可能位于哪类存储，并说明只是“源代码语义”还是编译器最终
决定：

```cpp
__shared__ float tile[32][32];
float accumulator = 0.0F;
float value = input[index];
output[index] = value;
```

再运行 `device_query`，记录：

- Global memory 容量。
- Shared memory per block。
- Shared memory per SM。
- Registers per block 和 per SM。

## 9. 本章小结

```text
Register 服务单个 thread。
Shared memory 服务一个 block 的协作。
Global memory 保存主要大规模数据。
性能优化经常是在减少、合并或复用数据移动。
```
