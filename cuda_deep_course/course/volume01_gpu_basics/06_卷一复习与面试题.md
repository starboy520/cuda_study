## 4. 计算题

### 题一

```text
blockDim = (16, 16, 1)
```

求 thread 总数和 warp 数。

答案检查：

```text
threads = 16 x 16 = 256
warps = 256 / 32 = 8
```

### 题二

```text
blockDim = (32, 8, 1)
threadIdx = (7, 3, 0)
```

求线性 thread ID：

```text
linear = y * blockDim.x + x
       = 3 * 32 + 7
       = 103
```

它属于：

```text
warp = floor(103 / 32) = 3
lane = 103 % 32 = 7
```

### 题三

```text
blockDim = (4, 3, 1)
blockIdx = (5, 2, 0)
threadIdx = (2, 1, 0)
```

求 block 内线性 thread ID：

```text
linearThreadId = 1 * 4 + 2 = 6
```

`blockIdx` 不参与这个计算，因为它只描述 thread 在自己 block 内排第几个。

如果矩阵：

```text
width = 100
```

则该 thread 负责的全局坐标是：

```text
col = 5 * 4 + 2 = 22
row = 2 * 3 + 1 = 7
matrixIndex = 7 * 100 + 22 = 722
```

所以 `linearThreadId=6` 和 `matrixIndex=722` 是两个不同概念。

## 5. 面试式回答

### 为什么 GPU 适合并行计算？

不要只回答“核心多”。完整回答应包含：

- 面向吞吐的设计。
- 大量并行执行资源。
- SIMT/warp 执行。
- 多个 warp 隐藏延迟。
- 适合大量相似且相对独立的工作。
- 性能仍取决于访存、控制流和数据传输。

### Block 和 Warp 有什么区别？

回答应包含：

- Block 是程序员定义的逻辑协作单元。
- Warp 是硬件组织 thread 的执行单位。
- block 可以包含多个 warp。
- block 内可以共享 shared memory 和进行 block 级同步。

### 为什么 block size 常取 32 的倍数？

因为 warp 包含 32 个 thread，使用 32 的倍数通常避免最后一个 warp 只有少量
有效 thread。但这只是起点，还要考虑资源、数据形状和性能实测。

## 8. 深入追问与答案提示

### CPU 也有 SIMD，为什么还需要 GPU？

回答应讨论规模、内存带宽、线程并行、延迟隐藏和工作负载，而不是否定 CPU
并行能力。

### 为什么一个 thread 不能等同于一个 CUDA Core？

Thread 是带有程序状态的逻辑实例；执行管线随时间处理许多 warp 指令。驻留
thread 数通常远多于某一时刻可执行的标量操作数。

### 为什么 block 不能迁移 SM？

因为它的 registers、shared memory 和 barrier 状态驻留于当前 SM，迁移会破坏
低成本调度模型。

### Occupancy 高但性能低，可能是什么原因？

- Warp 大多 stalled。
- Memory access 不合并。
- 已达到带宽上限。
- 指令依赖严重。
- 分歧或 barrier。
- 做了不必要工作。

### GPU kernel 很快但应用慢，下一步查什么？

用 Nsight Systems 检查传输、同步、CPU 空洞、launch 数、batch 和端到端占比。

> 这里集中放 §2（16 道概念题）和 §3（纠错题）的参考答案。§4 计算题、§5/§8 面试题
> 正文已带答案/提示。建议先盖住本节作答，再逐条对照。

### §2 核心概念检查 · 参考答案

1. **CPU vs GPU 设计目标**：CPU 面向**低延迟**（少量强核、大 cache、复杂乱序，让单个任务尽快完成）；GPU 面向**高吞吐**（海量简单核 + 大量线程，靠并行把总吞吐做大）。
2. **延迟 vs 吞吐**：延迟 = 单个任务从开始到完成的时间；吞吐 = 单位时间完成的任务总量。GPU 牺牲单任务延迟换总吞吐。
3. **加速部分快、端到端仍慢**：受 **Amdahl 定律**限制——没并行化的部分（H2D/D2H 传输、CPU 预处理、串行段）会锁死整体加速上限。kernel 再快，它占比小就没用。
4. **SM 是硬件还是逻辑**：**硬件**。Streaming Multiprocessor 是物理执行单元。
5. **Block 是硬件还是逻辑**：**逻辑**。由程序员定义的线程协作单元，由调度器映射到某个 SM。
6. **一个 block 能否跨多个 SM**：**不能**。一个 block 整体驻留在**单个 SM** 上（它的寄存器/shared/barrier 状态都在该 SM）。
7. **一个 SM 是否只能执行一个 block**：**否**。资源够时一个 SM 可**同时驻留多个 block**。
8. **一个 warp 多少 thread**：**32**。
9. **Thread 是否等同于 CUDA Core**：**不是**。thread 是带状态的逻辑执行实例；CUDA Core 是物理 ALU。驻留 thread 数远多于某一刻能执行的标量操作数（靠时分复用 + 延迟隐藏）。
10. **二维 block 必须正方形吗**：**不必**。可以是 `(32, 8)` 等任意形状（受每 block ≤1024 线程等限制）。
11. **`linearThreadId` 是全局 thread ID 吗**：**不是**。它只是 thread 在**自己 block 内**的线性编号，不含 `blockIdx`。
12. **`linearThreadId` vs 矩阵一维地址**：前者是"线程在 block 内排第几"（`y*blockDim.x+x`）；后者是"数据在矩阵中的位置"（`row*width+col`）。两者计算方式和含义都不同（见 §4 题三：6 vs 722）。
13. **register / shared / global 谁共享**：register = **单个 thread** 私有；shared = **同一 block** 内共享；global = **所有 thread**（整个 device）可见。
14. **为什么需要很多可运行 warp**：用于**延迟隐藏**——某些 warp 在等内存时，调度器切到其他就绪 warp 执行，让执行单元不空转。
15. **Occupancy 越高一定越快吗**：**不一定**。occupancy 只是"驻留并发 warp / 上限"，高它是为隐藏延迟；但若已达带宽上限、或 ILP 足够，低 occupancy 也可能很快，高也不保证最快。
16. **规格带宽 vs 实测带宽**：规格带宽是厂商理论峰值；实测带宽是真实 kernel 跑出来的有效带宽，受访问模式、合并、cache、负载影响，通常低于峰值。**别拿规格峰值当实测**。

### §3 常见错误说法 · 纠正

```text
❌ GPU 每个 thread 都独占一个 CUDA Core
✅ thread 远多于 CUDA Core，靠时分复用 + 延迟隐藏轮流使用执行单元；thread 是逻辑实例，core 是物理 ALU。

❌ 一个 block 固定对应一个 SM
✅ 一个 block 确实只在一个 SM 上执行，但一个 SM 可同时驻留多个 block；且哪个 block 去哪个 SM 由调度器决定，不是固定一对一。

❌ 二维 block 必须是 16x16 或 32x32
✅ block 形状灵活，只要满足每 block ≤1024 线程等硬件限制即可（如 32x8、256x1）。常取 32 倍数是为了对齐 warp，不是必须方形。

❌ Shared memory 比 global 快，所以用了就一定更快
✅ 只有存在复用/重排/协作时才值得用；无复用还硬上 shared，反而多一道 __syncthreads() 和复杂度，可能更慢。

❌ 线程越多，程序一定越快
✅ 超过隐藏延迟所需后，更多线程不再加速；还可能因寄存器/shared 资源压力降低 occupancy 或触发 spill 而变慢。

❌ Occupancy 达到 100%，性能就最好
✅ occupancy 高只代表并发 warp 多，不等于快。真正瓶颈可能是带宽、访存不合并、指令依赖、分歧等，需 profiler 确认。
```
