# 06 库的边界与算子工程化

> 前五章你学会了手写 GEMM、softmax、LayerNorm 并优化它们。但工业现实是：**大部分
> 时候你不该手写**——cuBLAS、cuDNN 已经把这些做到接近极致。本章讲清"什么时候用库、
> 什么时候手写"这条边界，以及算子开发的工程化（性能报告、代码审查、卷积的几种实现）。

## 1. 先明确：手写是为了理解，不是为了打败库

```text
你手写 GEMM 的价值：
  ✅ 理解 tiling/复用/Tensor Core 的原理（前几章）
  ✅ 能读懂 CUTLASS、能定制特殊算子
  ❌ 不是为了在标准 GEMM 上跑赢 cuBLAS（几乎不可能）

cuBLAS 是 NVIDIA 用汇编级优化 + 针对每种架构/尺寸调优的成果，
手写很难在"标准稠密 GEMM"上超过它。承认这一点不丢人，是工程成熟的标志。
```

## 2. 主要库的分工

| 库 | 领域 | 典型用途 |
|----|------|----------|
| **cuBLAS** | 稠密线性代数 | GEMM、GEMV、批量矩阵乘 |
| **cuBLASLt** | 轻量级 GEMM | 更灵活的 GEMM（融合 epilogue、混合精度） |
| **cuDNN** | 深度学习原语 | 卷积、池化、归一化、激活、RNN |
| **cuSPARSE** | 稀疏矩阵 | 稀疏 GEMM、SpMV |
| **cuFFT** | 快速傅里叶变换 | 信号处理、谱方法 |
| **CUTLASS** | GEMM 模板库 | 需要定制的 GEMM（开源、可改） |
| **Thrust** | 并行算法 | scan、sort、reduce（STL 风格） |

## 3. 决策树：用库还是手写

```text
你的算子是不是"标准操作"（稠密 GEMM、标准卷积、FFT）？
├─ 是 → 用库（cuBLAS/cuDNN）。手写大概率更慢且更花时间。
└─ 否 → 继续问：

是不是"访存受限的小算子"（norm、激活、bias、残差、elementwise）？
├─ 是 → 手写 + 融合。库的单算子调用会有多次 global 往返，
│       手写融合能省带宽（卷六第 05 章）。这是手写最有价值的场景！
└─ 否 → 继续问：

是不是"库不支持的特殊结构"（非标准稀疏、自定义 attention、奇怪的数据布局）？
├─ 是 → 手写 / 用 CUTLASS 定制 / 用 Triton。
└─ 否 → 优先用库，把精力花在更值钱的地方。
```

**核心判断**：

```text
计算受限的标准算子（GEMM/卷积）→ 用库（你赢不了）
访存受限的碎算子（norm/激活/融合）→ 手写（库的多次调用浪费带宽）
特殊结构 → 手写或 CUTLASS/Triton
```

这就是"库的边界"：**库擅长把单个标准算子做到极致，但跨算子的融合、特殊结构是手写
的天地。** 现代框架（PyTorch + Triton、TensorRT）正是"标准算子调库 + 碎算子融合手写"。

## 4. 卷积：三种实现路线（理解库在做什么）

卷积是 cuDNN 的核心，但理解它的几种实现能帮你判断何时手写：

### 4.1 直接卷积（direct）

```text
直接按定义滑窗：output[i] = Σ input[i+k] · filter[k]
- 用 shared memory 缓存输入 tile + halo（卷四第 05 章）
- 适合小 filter、特定形状
- 数据复用：相邻输出共享大量输入（filter 宽度倍的复用）
```

### 4.2 im2col + GEMM

```text
把卷积"摊平"成矩阵乘：
  im2col：把每个滑窗位置的输入展开成矩阵的一列
  然后 filter 矩阵 × 展开矩阵 = 卷积结果
优点：复用高度优化的 cuBLAS GEMM
缺点：im2col 展开后数据膨胀（重复存储重叠区），占显存、占带宽
这是很多框架的默认实现——把卷积变成 GEMM 来蹭 cuBLAS。
```

### 4.3 隐式 GEMM / Winograd / FFT

```text
implicit GEMM：不真正展开 im2col，边算 GEMM 边按卷积索引取数（cuDNN 常用）
Winograd：     用数学变换减少乘法次数（适合 3×3 等小 filter）
FFT 卷积：     频域相乘（适合大 filter）
cuDNN 会根据形状自动选最优算法（algorithm selection）。
```

> 理解这些让你明白：**cuDNN 不是一个算法，是一组算法 + 自动选择**。手写卷积要赢它，
> 得在特定形状上比它选的算法更优——很难，所以标准卷积用 cuDNN。

## 5. 算子工程化：性能报告

写完一个算子，要像卷五那样产出**可信的性能报告**，而不只是"我觉得快了"：

```text
一份算子性能报告应包含：
1. 正确性：和参考实现（CPU double / cuBLAS）的相对误差
2. 测试覆盖：多种尺寸（含非整除）、边界（n=1、大 K）
3. 性能：GFLOPS / 有效带宽 + 占峰值百分比
4. 对比基线：vs naive、vs cuBLAS/cuDNN（差距多少）
5. Roofline 落点：算力受限还是带宽受限
6. profiler 证据：Nsight Compute 的关键指标
7. 环境：GPU 型号、CUDA 版本、编译参数、输入分布
```

## 6. 算子代码审查（review）清单

手写算子提交前自查（也是面试/团队协作的标准）：

```text
正确性：
  [ ] 和参考实现对比（容差，不是 ==）
  [ ] 覆盖非整除尺寸、边界（n=1、超大、超小）
  [ ] compute-sanitizer memcheck/racecheck 通过

性能：
  [ ] 有效带宽/GFLOPS 算过，知道占峰值多少
  [ ] 访问合并（coalescing）确认
  [ ] shared memory 无意外 bank conflict
  [ ] occupancy 合理（不为凑 occupancy 牺牲算法）

工程：
  [ ] 边界判断完整，无越界
  [ ] 数值稳定（softmax 减 max、norm 的 eps）
  [ ] 资源正确释放，无泄漏
  [ ] 可复现的 benchmark 和测试
```

## 7. 本章小结（也是卷六总结）

```text
手写的价值：理解原理、能定制、能融合；不是为了打败 cuBLAS
库的分工：cuBLAS(GEMM)、cuDNN(卷积/DL)、cuSPARSE、cuFFT、CUTLASS、Thrust

决策边界：
  标准计算受限算子(GEMM/卷积) → 用库（赢不了）
  访存受限碎算子(norm/激活/融合) → 手写融合（库多次调用浪费带宽）← 手写主战场
  特殊结构 → 手写 / CUTLASS / Triton

卷积三路线：direct / im2col+GEMM / implicit GEMM·Winograd·FFT（cuDNN 自动选）
工程化：性能报告(正确性+GFLOPS+Roofline+profiler证据) + 代码审查清单
```

## 8. 卷六回顾：你学到了什么

```text
01 GEMM 数学/Roofline   → 为什么 naive 慢（AI=0.25），理论能多快（AI≈N/6）
02 Tiling/register      → shared/寄存器复用把 AI 推向拐点
03 向量化/双缓冲/TensorCore → 逼近峰值，专用单元放大算力
04 Softmax              → 减最大值防溢出 + 归约结构 + online softmax
05 LayerNorm/融合       → 归约结构 + 融合省带宽（访存受限优化核心）
06 库的边界             → 何时用库、何时手写、算子工程化
```

主线一以贯之：**用数据复用提升算术强度（计算受限算子），用融合减少访存（访存受限
算子），用库站在巨人肩上，用手写解决库覆盖不到的地方。**
