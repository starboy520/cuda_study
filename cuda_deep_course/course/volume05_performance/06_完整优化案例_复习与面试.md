# 06 完整优化案例、复习与面试

## 1. 案例：Transpose

### Assess

Naive transpose：

- 读 input 连续。
- 写 output 跨步。
- 计算量极低。

假设：kernel memory-bound，主要问题是 global store 不合并。

### Parallelize

Naive 版本先通过 CPU reference、非方阵和非整除测试。

### Optimize

引入 shared tile：

```text
合并读
-> shared 重排
-> 合并写
```

再加入 padding 降低 bank conflict。

### Evidence

需要同时比较：

- 时间和有效 GB/s。
- Global store sectors/efficiency。
- Shared bank conflict。
- Sanitizer。

若时间没有改善，应检查输入规模、cache、边界比例、profiling overhead 和测试
环境，而不是直接宣布理论错误。

## 2. 案例：Reduction

优化阶梯：

```text
global atomic
-> block shared tree
-> 每 thread 多元素
-> warp shuffle 收尾
```

每一步提出一个单一假设：

- 减少 global atomic 竞争。
- 减少 block 数和 global load。
- 减少 barrier/shared 操作。

使用大规模输入验证，避免 launch latency 主导。

## 4. 常见反模式

- 没有 CPU reference 就优化。
- 只测一次。
- 只报告 speedup，不报告 baseline。
- 用 CPU timer 却不等待 GPU。
- 看到 occupancy 低就盲目提高。
- 看到 memory throughput 高就认为必须降低。
- 同时改很多因素，无法解释收益。
- 用 profiler 运行时间当正常 benchmark 时间。
