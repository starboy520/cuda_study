# 06 完整优化案例、复习与面试

## 0. 本章定位

前五章把工具和指标逐个讲过了，本章把它们**串成一条完整的优化叙事**——这正是面试官
最想听的：给你一个 kernel，你怎么一步步从"能跑"做到"快且有据可依"。读完你应该能对着
transpose 和 reduction，用 3 分钟讲清"问题→假设→改法→证据→结论"的闭环。

## 1. 案例：Transpose（矩阵转置）

转置是最经典的 **memory-bound 入门案例**：计算量几乎为零（只是搬数据），所以它的性能
**完全由访存效率决定**，是练"合并访问 / shared memory / bank conflict"三件套的最佳载体。

### 1.1 Assess：先定性，判断瓶颈在哪

naive 转置长这样：

```cpp
// naive：out[x][y] = in[y][x]
int x = blockIdx.x * blockDim.x + threadIdx.x;
int y = blockIdx.y * blockDim.y + threadIdx.y;
out[x * width + y] = in[y * width + x];   // 读连续，写跨步
```

逐项评估：

```text
计算量：     每个元素只有一次赋值，0 次浮点运算 -> AI 极低 -> 必然 memory-bound
读 in：      同一 warp 的 32 条 lane 读 in[y*width + x]，x 连续 -> 合并访问 ✅
写 out：     同一 warp 写 out[x*width + y]，相邻 lane 的 x 相差 1，地址相差 width
            -> 跨 width 的大跨步 -> 完全不合并 ❌
```

**假设（一句话）**：kernel 是 memory-bound，瓶颈在 **global store 不合并**——每个 warp
的 32 个写分散到 32 个不同的 cache line/sector，写带宽利用率只有理论的 1/32 量级。

> 这一步只用"读代码 + 访存模式分析"就能下假设，**还没碰 profiler**。但假设必须后面用
> 数据验证（1.4 节），否则只是猜。

### 1.2 Parallelize：先保证正确，再谈快

优化前必须有一个**可信的正确基线**，否则后面所有"变快"都可能是把活算错了换来的：

```text
1. 写 CPU reference transpose，逐元素比对。
2. 专门测非方阵（如 1003 x 769）：能抓出把 width/height 写反的 bug。
3. 测非整除规模（如 1003 x 769 对 32x32 tile 不整除）：验证边界 if 防护。
4. compute-sanitizer --tool memcheck 跑一遍：确认没有越界写。
```

naive 版先通过这四关，才有资格作为优化的 baseline。

### 1.3 Optimize：用 shared tile 把"不合并的写"换成"合并的写"

核心思路：**不要直接跨步写 global，先在 shared memory 里完成转置，再让 global 的读和写
都保持合并**。

```text
第1步  合并读：warp 连续读 in 的一行，写进 shared tile（按行存）
第2步  __syncthreads()：等整个 tile 到位
第3步  合并写：warp 连续写 out 的一行，数据从 shared tile 按列取
       —— 转置发生在 shared 内部（读行/取列），global 两端都合并
```

```cpp
__shared__ float tile[32][32 + 1];   // 注意 +1，见下
int x = blockIdx.x*32 + threadIdx.x;
int y = blockIdx.y*32 + threadIdx.y;
tile[threadIdx.y][threadIdx.x] = in[y*width + x];   // 合并读
__syncthreads();
// 用转置后的坐标写回，warp 内连续 -> 合并写
int tx = blockIdx.y*32 + threadIdx.x;
int ty = blockIdx.x*32 + threadIdx.y;
out[ty*height + tx] = tile[threadIdx.x][threadIdx.y];
```

**为什么 `tile[32][32 + 1]` 要加那个 +1（padding）？** 这是 bank conflict 的经典修复。
shared memory 分 32 个 bank，按 4 字节交错。若 tile 宽度正好是 32，则 `tile[0][k]`、
`tile[1][k]`…`tile[31][k]`（同一列）全部落在**同一个 bank**——第 3 步按列取数时，一个
warp 的 32 条 lane 撞同一 bank，被串行化成 32 次访问（32-way conflict）。把宽度改成 33，
列方向的地址每行错开一个，32 个元素正好散到 32 个不同 bank，冲突消除。

```text
宽度 32（坏）：列内 32 个元素 -> 同 1 个 bank -> 32-way conflict，串行 32 次
宽度 33（好）：+1 让每行偏移 1 -> 32 个元素散到 32 个 bank -> 0 conflict
```

### 1.4 Evidence：用数据验证每一条假设

优化做完不能只说"我加了 shared tile"，要拿出证据链，逐条对上 1.1 的假设：

| 指标 | 看它验证什么 | 期望变化 |
|---|---|---|
| 时间 + 有效 GB/s | 整体是否变快、离带宽峰值多远 | GB/s 显著上升，逼近 T4 峰值 |
| `gst_efficiency`（global store 效率）| 1.1 的核心假设：写不合并 | naive 接近 1/32；shared 版接近 100% |
| `gld_efficiency`（global load 效率）| 读一直是合并的 | 两版都应接近 100% |
| shared bank conflict 计数 | padding 是否真的消除冲突 | 加 +1 后应降到 ~0 |
| sanitizer | 正确性兜底 | memcheck/racecheck 全过 |

**如果时间没改善怎么办？**（这是面试高频追问）按顺序排查，而不是立刻怀疑理论：

```text
1. 输入规模太小？ kernel 几微秒，launch 开销和噪声淹没差异 -> 加大到 4096x4096
2. 边界 tile 占比太高？ 非整除规模下大量半空 tile 摊薄收益
3. 把 profiler 运行时间当 benchmark？ profiler 有 overhead，另测纯时间
4. cache 帮了 naive？ 小矩阵整个塞进 L2，naive 的跨步写被 cache 吸收
5. 没 warmup？ 首次含 context/JIT 开销
```

## 2. 案例：Reduction（归约）

完整推导见卷四第 03 章，这里只从**"每一步是一个可验证的假设"**这个面试视角复述优化阶梯：

```text
v1 global atomic     -> v2 block shared tree -> v3 每 thread 多元素 -> v4 warp shuffle 收尾
```

每一步都对应**单一假设 + 单一指标**，这正是面试要的"控制变量"思维：

| 版本 | 单一假设（这步要解决什么）| 期望指标变化 |
|---|---|---|
| v1→v2 | 全局原子让所有线程争一个地址，串行化 | atomic 竞争消失；DRAM throughput 上升 |
| v2→v3 | 第一轮树归约只有半数线程活跃，浪费 | block 数减半；global load 利用更满 |
| v3→v4 | `offset<32` 后整 block barrier 是多余开销 | barrier/`__syncthreads` 次数下降；shared 流量下降 |

三条纪律（面试官爱听）：

- **必须用大规模输入验证**（如 `16777216`）。小输入上 launch latency 主导，shuffle 的
  收益会被噪声淹没，甚至看起来更慢。
- **先正确后性能**：每版都要在 `n=1, 31, 257, 1000003` 这些刁钻规模上 PASS。
- **每步只改一处**：同时改两个优化，就无法把性能变化归因到哪一个。

> 归约是 memory-bound：理论下界就是"把输入读一遍"的时间。优化的终点不是"无限快"，而是
> **第一阶段 kernel 的 DRAM 带宽逼近硬件峰值**——到了带宽墙就该停。

## 4. 常见反模式

- 没有 CPU reference 就优化。
- 只测一次。
- 只报告 speedup，不报告 baseline。
- 用 CPU timer 却不等待 GPU。
- 看到 occupancy 低就盲目提高。
- 看到 memory throughput 高就认为必须降低。
- 同时改很多因素，无法解释收益。
- 用 profiler 运行时间当正常 benchmark 时间。

## 5. 面试题（附参考答案）

下面每题给出"一句话答案 + 关键补充"。面试时先给一句话结论，再按需展开。

**1. 如何判断 kernel 是 memory-bound 还是 compute-bound？**
一句话：看 Nsight Compute 的 **Speed of Light**——内存吞吐和计算吞吐哪个更接近 100%，
就被哪个限制。
补充：也可用 Roofline——算出 kernel 的 arithmetic intensity（AI），若 AI 在拐点左边
（低于 `peak FLOPS / peak BW`）就是 memory-bound。transpose、vector add、reduction 几乎
都是 memory-bound；大 tile 的 GEMM 才可能 compute-bound。

**2. 有效带宽如何计算？**
一句话：`有效带宽 = 算法必须读写的字节数 / 耗时`。
补充：transpose 至少读 N + 写 N = 2N 字节，`GB/s = 2N / time / 1e9`。注意这是"算法逻辑
字节"，不等于硬件实际 DRAM 事务字节（不合并/重复加载会让实际事务更多）。拿它和 T4 峰值
（约 320 GB/s）比，就知道还有多少空间。

**3. Arithmetic intensity 是什么？**
一句话：每从某层内存搬 1 字节，能做多少次浮点运算（FLOP/byte）。
补充：它是 Roofline 的横轴。AI 低 → memory-bound；AI 高 → 才有可能 compute-bound。注意要
说明"字节"是相对哪一层（DRAM/L2/shared），同一 kernel 对不同层 AI 不同。

**4. Roofline 有什么局限？**
一句话：它只给**上限**，不保证 kernel 能贴到线；且是简化模型。
补充：它不建模 latency、occupancy 不足、控制分歧、cache 命中波动等。AI 也依赖你选哪层
内存。它适合判断"还有多少理论空间 + 该往内存还是计算方向优化"，不适合精确预测。

**5. Occupancy 低一定慢吗？**
一句话：不一定。
补充：occupancy 只衡量"驻留并发 warp 数 / 上限"，高 occupancy 是为了**用足够多 warp 隐藏
内存延迟**。但若每个线程做更多独立工作（ILP 高）、或本就 memory-bound 已打满带宽，低
occupancy 也能很快（经典反例：Volkov 的低占用高性能 GEMM）。反过来 occupancy 100% 也不
保证最快——带宽可能才是墙。

**6. Nsight Systems 与 Nsight Compute 有何区别？**
一句话：Systems 看**系统级时间线**（找哪段慢、谁在等谁），Compute 深挖**单个 kernel 的
微架构指标**。
补充：流程是先用 Systems 定位"时间花在哪个阶段/是否有空洞/传输和计算是否重叠"，再用
Compute 对锁定的 kernel 分析 SoL、内存、occupancy、stall。先系统、后 kernel。

**7. 如何正确测量异步 kernel？**
一句话：用 **CUDA Event** 记录在同一 stream 上，`cudaEventSynchronize(stop)` 后再读时间；
若用 CPU timer，结束前必须 `cudaDeviceSynchronize()`。
补充：kernel launch 是异步的，不同步只会测到 launch 开销。还要先 warmup、跑多次取中位数、
明确区间是否含 H2D/D2H。

**8. 为什么需要 warmup？**
一句话：首次运行包含 context 初始化、module 加载/JIT、内存首触、频率爬升等一次性开销，
会污染稳态测量。
补充：稳态吞吐要在 warmup 之后测；但首次延迟本身若是业务指标（如推理冷启动），应**单独
报告**，不能简单丢。

**9. Warp divergence 发生在哪个范围？**
一句话：**warp 内部**（32 条 lane 走不同路径）才算 divergence；不同 warp 走不同路径无代价。
补充：同 warp 的 lane 共用一个 PC，分歧时各路径**串行**执行、被关闭的 lane 空转，最坏
32 路 switch 慢约 32 倍。优化方向是让分歧落在 warp 之间（按条件排序数据、对齐分支边界）。

**10. Register pressure 如何影响性能？**
一句话：寄存器用多了，每 SM 能驻留的 warp 变少（occupancy 下降）；但强行压低又会 **spill**
到 local memory 反而更慢。
补充：用 `nvcc -Xptxas=-v` 看每线程寄存器数和 spill。要在"足够寄存器避免 spill / 足够
occupancy 隐藏延迟"之间权衡，`__launch_bounds__` / `--maxrregcount` 可调，但必须实测。

**11. 如何验证一次优化确实解决了目标瓶颈？**
一句话：优化前先**预测"哪个指标会变"**，改完用 profiler 确认**正是那个指标**按预期变化、
且时间随之改善。
补充：例：假设"写不合并"，就预测 `gst_efficiency` 上升；若时间快了但该指标没动，说明你
其实改对了别的东西，假设没被验证。一次只改一处，才能归因。

**12. 为什么 profiler 与普通运行时间不同？**
一句话：profiler 为采集硬件计数器会**插桩、可能重放 kernel、串行化**，引入显著 overhead。
补充：所以 profiler 的 kernel 时间不能当 benchmark 数字用；要分开做——干净环境测时间，
profiler 单独看指标。Nsight Compute 尤其会多次重放同一 kernel 收集不同 section。
